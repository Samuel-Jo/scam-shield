문제 정의(DAY1~2)부터 최종 발표(DAY16)까지, 역할을 나눠 함께 달립니다.

## 2. 팀원 R&R (역할별 담당 DAY / 임무)

| 역할 | 이름 | 담당 DAY | 핵심 임무 |
|------|------|----------|-----------|
| PM / 기획 (팀장) | 이도윤 | DAY 0~16 전체 | 일정 조율, 문제 정의(DAY1~3), PRD 작성·갱신, 의사결정 조율, DAY16 발표 구성 |
| 개발 | 박서준 | DAY 5~14 중심 | 규칙 엔진 구현(DAY5~6), TRD 작성·갱신, Harness 코드, DAY14 정식 배포 |
| 평가 / 데이터 | 최유나 | DAY 6~15 중심 | 평가 케이스 16개 설계(DAY6~7), Pass Rate 측정, 가드레일 검증, final_report.md |
| UX / 검증 | 정하린 | DAY 3~16 중심 | 사용자 시나리오(DAY3~4), 중간 데모 진행(DAY12), 피드백 정리, DAY16 시연 |

### R&R 경계 원칙

- **PM**: 범위(Scope) 수호자. 기능이 추가되려면 PM이 PRD를 먼저 갱신한다.
- **개발**: TRD 갱신 없이 구조 변경 금지. 결정이 필요하면 ADR 작성.
- **평가**: 수치 없이 "좋아졌다"는 표현 금지. 개선은 Pass Rate로 증명한다.
- **UX**: 사용자 검증 없이 "쉽다"는 표현 금지. 테스터 반응을 반드시 기록한다.

## 3. Docs as Code 폴더 구조

> "Docs as Code"란, 기획 문서도 코드처럼 Git으로 버전을 관리하는 방식입니다.
> 마치 교과서를 고칠 때 수정 이력을 남기는 것처럼, 문서 변경 내역이 모두 남습니다.

| 폴더 / 파일 | 설명 | 시작 시점 |
|------------|------|-----------|
| `docs/` | 산출물 .md 파일 모음 (기획·설계·평가 결과) | DAY0 |
| `src/` | 규칙 엔진·UI 코드 | DAY5부터 |
| `tests/` | 평가 케이스 JSON | DAY7부터 |
| `JOURNEY.md` | DAY별 진행 일지 | DAY0 |
| `CHARTER.md` | 팀 헌장 (이 파일) | DAY0 |
| `README.md` | 저장소 소개 | DAY0 초안 → DAY14 정식화 |

### docs/ 파일은 번호 접두어로 시간 순서를 표시합니다

예시: `01-problem_A.md`, `04-prd.md`, `15-final_report.md`
자세한 예고 목록은 `docs/00-folder-structure.md`를 참고하세요.

## 4. 브랜치 전략

> 브랜치(branch)란 "작업 공간을 복사해서 따로 작업하는 방법"입니다.
> 마치 시험지를 복사해 연습하다가 완성되면 원본에 붙이는 것과 같습니다.

### 규칙

- `main` 브랜치는 **항상 작동하는 안정 버전**입니다. 직접 push(저장) 하지 않습니다.
- 새 작업을 시작할 때는 반드시 `feature/기능명` 브랜치를 만들어 작업합니다.
- 작업이 끝나면 PR(Pull Request)을 만들고, 팀원 1명 이상이 확인한 뒤 `main`에 합칩니다.

### 브랜치 이름 예시

| 상황 | 브랜치 이름 |
|------|------------|
| PRD 초안 작성 | `docs/prd-v1` |
| 규칙 엔진 구현 | `feature/rule-engine-v1` |
| PII 마스킹 추가 | `feature/pii-masking` |
| 최유나의 평가 케이스 작업 | `feature/eval-cases` |

### GitHub Desktop으로 브랜치 만드는 방법

1. GitHub Desktop 상단 "Current Branch" 클릭
2. "New Branch" 버튼 클릭
3. 이름 입력 (예: `docs/prd-v1`) → "Create Branch" 클릭
4. 작업 후 "Commit to docs/prd-v1" → "Push origin" 클릭
5. "Create Pull Request" 버튼으로 PR 생성

## 5. 커밋 컨벤션

> 커밋(commit)이란 "저장 + 메모"입니다.
> 말머리(접두어)를 달면 나중에 어떤 변경인지 바로 알 수 있습니다.
> Cursor AI의 자동요약 기능을 활용해 메시지를 만들어도 됩니다.

### 접두어 규칙

| 접두어 | 뜻 | 언제 쓰나 |
|--------|-----|----------|
| `feat:` | 새 기능 추가 | 코드에서 새 기능이 생겼을 때 |
| `docs:` | 문서 작성·수정 | .md 파일을 새로 쓰거나 고쳤을 때 |
| `fix:` | 버그·오류 수정 | 잘못된 내용이나 오류를 고쳤을 때 |

### 커밋 메시지 예시 3개

```
docs: DAY0 팀 헌장(CHARTER.md) 초안 작성
feat: 규칙 엔진 V1 — 키워드 7종 탐지 로직 추가
fix: PRD v1 — 성공 기준 수치 오탈자 수정
```

## 7. 의사결정 규칙

1. 기술·UX 선택처럼 **각 담당자 범위 내의 결정**은 담당자가 판단합니다.
2. 범위 변경·일정 영향이 있는 결정은 **PM(이도윤)이 조율**하고 팀 합의를 거칩니다.
3. 팀 합의가 이루어지지 않을 때는 **advisor(기대표)에게 에스컬레이션**합니다.
4. 중요한 결정은 반드시 `JOURNEY.md`의 "결정" 항목에 기록합니다.
