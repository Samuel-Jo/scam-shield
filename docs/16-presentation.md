# AI 사기 메시지 방패

> "사기 문자를 받은 순간, 누구나 3초 안에 위험을 알아챈다."

**팀 구성**
- 이도윤 (PM) — 프로젝트 방향·의사결정
- 박서준 (개발) — 탐지 엔진·CLI·가드레일 구현
- 최유나 (평가/데이터) — 평가셋 설계·Pass Rate 측정
- 정하린 (UX) — 사용자 시나리오·데모·아키텍처 시각화

---

## 1. Problem — 누구의, 어떤 문제인가

**대상 사용자**
스마트폰 문자나 중고거래 채팅을 이용하는 일반인. 특히 "이게 사기야?" 판단이 어려운 첫 중고거래 이용자, 디지털 경험이 적은 분들.

**지금 어떻게 하고 있나 (Before)**
- 카카오톡 단체방에 스크린샷을 올리고 답변을 기다립니다. 빠른 답변이 오지 않기도 하고, 사람마다 의견도 다릅니다.
- 포털에 문자 내용을 검색하지만, 신종 사기 수법은 검색 결과에 없는 경우가 많습니다.
- "의심은 되는데 확신이 없는" 불안 상태가 30초~10분 이상 지속됩니다.
- 결국 무서워서 정상 안내 문자도 무시하거나, 잘못 판단해 링크를 눌러 피해를 입기도 합니다.

**왜 AI로 풀기 적합한가**
1. 언어 패턴·맥락을 읽는 작업입니다. 사람마다 판단이 다른 "이 문장 사기야?"를 일관된 기준으로 처리할 수 있습니다.
2. 매일 수많은 사람이 같은 고민을 반복합니다. AI가 대신해주면 효율적입니다.
3. "사기 / 정상" 두 라벨로 정답이 명확합니다. Pass Rate로 개선을 숫자로 증명할 수 있습니다.
4. 텍스트 입력·분류·경고 출력 범위로 16일 안에 완주 가능합니다.

## 2. Demo — 서비스 작동 흐름

**실행 명령어**
```
node src/cli.mjs "문자 내용을 여기에 붙여넣으세요"
```

**화면 흐름 (단계별)**

```
[사용자]
  의심 문자 텍스트를 복사해서 CLI에 붙여넣는다
      ↓
[가드레일: guardInput]
  빈 입력 거부 / 5,000자 초과 자름 / PII 마스킹(전화·계좌번호 가운데 ***로 치환)
      ↓
[가드레일: detectPromptInjection]
  "이전 지시 무시" 같은 조작 패턴 탐지 → 경고 플래그
      ↓
[탐지 엔진: detectScam]
  7종 신호 스캔 (외부 링크·개인정보 수집·긴급성·금전·기관 사칭·당첨·비정상 발신)
  + 고위험 키워드 보너스 + 콤보 보너스 5종 → 위험도 점수 0~100 계산
      ↓
[가드레일: guardOutput]
  단정 표현 완화 / 면책 문구 부착 / 오탐 한계 안내
      ↓
[결과 출력]
  위험도 점수 / 위험 신호 목록 / 쉬운 한 줄 경고 / 추천 행동 안내
```

**출력 예시 (사기 문자)**

```
위험도: 87점 (위험)
감지된 신호: 외부 링크 유도, 긴급성 조성, 금전 요구
경고: 이 메시지는 외부 링크 클릭을 유도하고 있습니다. 즉시 판단이 필요합니다.
추천 행동: 링크를 누르지 마세요. 의심되면 금융감독원(1332) 또는 경찰청(112)에 신고하세요.
[안내] 일부 정상 문자가 오탐될 수 있습니다. 최종 판단은 본인이 내려주세요.
```

**5가지 핵심 기능 요약**
1. 위험도 점수 (0~100)
2. 위험 신호 하이라이트 + 이유
3. 쉬운 한 줄 경고
4. 추천 행동 안내 (신고 전화번호 포함)
5. 가드레일 4종 (PII 마스킹·인젝션 탐지·입력 검사·출력 검증)

- 두 케이스 모두 **맥락 이해**가 필요합니다. V3에서 LLM 보강으로 다룰 예정입니다.

## 4. Iteration — 16일 동안 어떻게 발전했나

**버전별 주요 변화**

```
[V1 · DAY 5]
  규칙 기반 탐지 엔진 최초 구현
  7종 신호 + 가중치 합산 + CLI 실행
  Pass Rate: 75% (12/16)
      ↓
[V2 · DAY 8~9]
  고위험 키워드 티어(KB) 추가
  콤보 보너스 5종 (V1 2종 → 5종)
  임계값·기존 가중치는 동결 (과적합 방지)
  Pass Rate: 90% (18/20)
      ↓
[가드레일 추가 · DAY 10]
  PII 마스킹 / 프롬프트 인젝션 탐지
  입력 검사 / 출력 검증 4종
  Pass Rate 90% 유지 확인
      ↓
[행동 안내 추가 · DAY 13]
  DAY 12 피드백 "그래서 어떻게 해야 하나" 반영
  추천 행동 한 줄 (신고 전화번호 포함)
  오탐 한계 안내 문구 추가
```

**DAY 12 피드백 반영 흐름**

| 피드백 | 중요도 | DAY 13 반영 여부 |
|--------|--------|----------------|
| "그래서 어떻게 해야 하나가 안 나와요" | 높음 | 반영 (행동 안내 추가) |
| "드라마 대화인데 위험하다고 나왔어요" | 높음 | 반영 (오탐 한계 안내) |
| "결과가 색으로 구분됐으면 좋겠어요" | 중간 | 차기 백로그 |
| "예시 버튼이 있으면 좋겠어요" | 낮음 | 차기 백로그 |

**의견 조율에서 배운 것**

1. **채점 가능성으로 문제 선택** (DAY 2): 정하린이 후보 B(공지 변환)에 미련이 있었지만, 최유나가 "16일 안에 Pass Rate로 개선을 측정할 수 있는 문제여야 한다"는 근거를 제시해 후보 A로 수렴했습니다. 좋은 아이디어도 측정할 수 없으면 증명이 어렵습니다.

2. **과적합 경계** (DAY 7): 최유나가 "지금 당장 규칙을 손봐 75%를 올리자"고 했을 때, 박서준이 "케이스에만 최적화하면 실제 세상에서 더 틀릴 수 있다"며 기준선 동결을 주장했습니다. 시험 문제에만 맞춘 공부가 응용에는 약한 것과 같은 원리입니다.

3. **PII 마스킹 절충** (DAY 10): 정하린은 식별성을, 최유나는 안전을 우선했습니다. "가운데만 가리고 끝자리는 남긴다"는 절충안으로 둘을 모두 챙겼습니다.

4. **한계 정직히 병기** (DAY 15): 정하린이 "전/후를 좋게 포장하자"고 했을 때, 최유나가 "수치와 한계를 있는 그대로 써야 신뢰를 얻는다"고 맞섰습니다. 이도윤이 "성과는 분명히, 한계도 정직히"로 합의를 이끌었습니다. 장기적으로는 솔직한 한계 인정이 더 높은 신뢰를 만듭니다.
