# 가드레일 구성 — 서비스를 지키는 안전장치 (DAY10)

> DAY 10 · AI 사기 메시지 방패 팀 · 담당: 박서준(개발) / 최유나(평가) · 문서 정리: 콘텐츠 라이터

---

## 왜 필요한가

서비스가 동작한다고 해서 안전한 것은 아닙니다. AI와 규칙 엔진을 다루는 서비스에는 세 가지 위험이 항상 존재합니다.

1. **AI 오작동**: 결과가 틀리거나 과도하게 단정적인 표현을 출력할 수 있습니다.
2. **악용 시도**: 악의적인 입력으로 시스템을 오동작시키려는 사람이 있을 수 있습니다.
3. **개인정보 노출**: 사용자가 붙여넣은 메시지에 전화번호·계좌번호 같은 민감한 정보가 포함될 수 있고, 이것이 결과 화면에 그대로 드러나면 안 됩니다.

가드레일은 이 세 위험을 막기 위해 **입력 전**, **출력 전**, **도구 사용 시점**에 각각 세워두는 안전장치입니다.

---

## Pre-LLM (입력 검사)

> 담당 함수: `guardInput` (`src/guardrails.mjs`)

입력이 규칙 엔진으로 넘어가기 전 단계에서 이상한 입력을 걸러냅니다.

| 검사 항목 | 동작 |
|----------|------|
| 빈 입력 | 빈 문자열 또는 공백만 있는 입력 → 즉시 거부, 안내 메시지 반환 |
| 초과 길이 | 5,000자를 초과하는 입력 → 5,000자에서 잘라냄 (처리 오남용 방지) |

**왜 5,000자인가**: 사기 문자는 대부분 짧습니다. 수만 자짜리 입력은 정상 사용이 아닐 가능성이 높고, 처리 시간과 자원을 낭비합니다. 5,000자는 충분한 메시지 길이를 허용하면서 오남용을 막는 균형점입니다.

---

## Post-LLM (출력 검증)

> 담당 함수: `guardOutput` (`src/guardrails.mjs`)

규칙 엔진이 결과를 계산한 뒤, 사용자에게 보내기 직전에 결과를 확인합니다.

| 검증 항목 | 동작 |
|----------|------|
| 단정 표현 완화 | "100% 사기" 같은 절대적 단정 표현을 "사기 가능성이 높습니다" 수준으로 완화 |
| 면책 문구 부착 | 모든 결과 하단에 고정 문구 한 줄 자동 추가 |

고정 면책 문구:

> "이 결과는 참고용입니다. 최종 판단은 사용자 본인이 하며, 확실하지 않을 경우 공식 기관에 확인하세요."

**왜 단정 표현을 완화하는가**: 규칙 기반 엔진은 맥락을 이해하지 못합니다. 같은 키워드가 사기 문자에도, 일상 대화에도 등장할 수 있습니다. "100% 사기"라는 표현은 서비스 신뢰성을 오히려 해칩니다. 정직한 불확실성 표현이 더 안전합니다.

## Tool Guardrail

V1~V2 전 구간에서 외부 API·서버 호출이 **0건**입니다. 규칙 엔진은 로컬에서만 동작하며, 인터넷 연결이 없어도 실행됩니다.

외부 호출이 없다는 것은 **공격면(attack surface)이 없다**는 뜻입니다. 외부 서비스가 없으니 API 키 탈취, 외부 서버 장애, 호출 제한 초과 같은 문제가 원천적으로 발생하지 않습니다.

> V2 이후 LLM 외부 호출을 도입할 경우 API 키 노출 방지·호출 횟수 제한(rate limit)을 별도 검토합니다.

## PII 마스킹 (필수)

> 담당 함수: `maskPII` (`src/guardrails.mjs`)

PII(Personal Identifiable Information)는 전화번호·계좌번호·주민등록번호·카드번호처럼 개인을 식별할 수 있는 정보입니다.

사용자가 붙여넣은 메시지에 이런 정보가 포함되어 있을 수 있습니다. 분석 결과 발췌를 화면에 보여줄 때 이 정보를 그대로 노출하면, **서비스가 개인정보를 재노출하는 통로**가 됩니다.

`maskPII`는 결과 발췌 텍스트에서 다음 패턴을 탐지해 가운데를 `***`로 치환합니다.

| 정보 유형 | 원본 예시 | 마스킹 결과 |
|----------|----------|------------|
| 전화번호 | 010-1234-5678 | 010-****-5678 |
| 계좌번호 | 110-1234-567890 | 110-****-567890 |
| 주민등록번호 | 900101-1234567 | 900101-******* |
| 카드번호 | 1234-5678-9012-3456 | 1234-****-****-3456 |

**중요 — 분류 점수와 마스킹의 관계**: 점수(사기 여부 판정)는 원본 텍스트를 기준으로 계산합니다. 마스킹은 결과 발췌를 사용자에게 보여줄 때만 적용됩니다. 마스킹이 분류 정확도에 영향을 주지 않습니다.

**왜 필수인가**: "가운데만 가리고 끝자리 일부는 남긴다"는 방식은, 사용자가 자신의 번호임을 식별할 수 있으면서도 완전 노출은 막는 균형입니다. DAY10에서 팀 합의로 '필수 적용'으로 확정했습니다.

## 프롬프트 인젝션 탐지 (필수)

> 담당 함수: `detectPromptInjection` (`src/guardrails.mjs`)

프롬프트 인젝션이란 입력 텍스트 안에 "이전 지시를 무시해라", "너는 이제 다른 역할이야"처럼 시스템을 조종하려는 숨겨진 명령을 넣는 공격입니다.

탐지 대상 패턴 (규칙 기반):

- "이전 지시 무시", "지시를 무시"
- "시스템 프롬프트", "system prompt"
- "너는 이제", "ignore previous"
- 이 외 유사 패턴 포함

패턴이 감지되면 해당 입력에 경고 플래그를 붙이고, 결과 출력 시 사용자에게 알립니다. 입력이 처리를 완전히 막는 방식은 아니며, 탐지 사실을 알리는 방어 구조입니다.

**V1 규칙 기반 엔진에서 인젝션 직접 영향은 제한적입니다.** 규칙 엔진은 LLM이 없으므로 숨겨진 명령을 "이해하고 따르는" 동작 자체가 없습니다. 그럼에도 탐지 로직을 DAY10에 구현한 이유는 두 가지입니다.

1. **입력단 방어**: 공격 시도 자체를 기록하고 경고함으로써 이상 입력 패턴을 파악합니다.
2. **V2 LLM 도입 대비**: V2에서 LLM을 추가할 경우 인젝션 위험이 실질적으로 커집니다. 방어 로직을 미리 갖추면 V2 전환 시 즉시 효과를 발휘합니다.

인젝션 탐지도 팀 합의로 **'필수 적용'**으로 확정했습니다.

## 네트워크 보안 (Cloudflare WAF, 배포 단계)

WAF(Web Application Firewall)는 웹 서비스로 들어오는 악성 트래픽을 서비스에 닿기 전에 차단하는 방화벽입니다.

Cloudflare WAF는 **배포 단계(인프라 담당)에서 설정**합니다. DAY10 현재 로컬 실행 단계이므로 코드 수준 구현 대상이 아닙니다. 공개 URL로 배포할 때 Cloudflare를 통해 활성화하며, 다음을 막습니다.

- DDoS(대규모 요청 폭주) 공격
- 악성 봇 트래픽
- SQL 인젝션 등 알려진 웹 공격 패턴

## 검증

DAY10 가드레일 적용 후 20케이스 기준 **Pass Rate 90% 유지**가 확인되었습니다.

| 항목 | 결과 |
|------|------|
| 전체 Pass Rate | 90.0% (18/20) — 가드레일 적용 전후 동일 |
| 마스킹 동작 | 전화·계좌·주민·카드번호 가운데 `***` 치환 정상 확인 |
| 인젝션 경고 | 탐지 대상 패턴 입력 시 경고 플래그 정상 출력 확인 |
| 분류 점수 영향 | 없음 — 마스킹은 발췌 표시 단계에만 적용, 점수 계산은 원본 기준 |

PII 마스킹과 인젝션 탐지는 분류 정확도에 영향을 주지 않습니다. 가드레일은 서비스 안전성을 높이면서도 탐지 성능을 그대로 유지합니다.
