## 1. PRD가 왜 필요한가 — "무엇을 만들지" 먼저 합의하기

PRD는 **Product Requirements Document**, 우리말로는 **제품 요구사항 문서**입니다. 어렵게 들리지만 핵심은 하나입니다.

> "우리 팀이 만들 것"을 한 문서에 써 놓아서, 기획·개발·디자인이 각자 다른 것을 만드는 사태를 막는다.

예를 들어 PRD 없이 시작하면 이런 일이 생깁니다.

- 개발자는 "로그인 기능도 넣자"며 3일을 씁니다.
- 디자이너는 "공유 버튼 디자인도 필요하다"며 화면을 그립니다.
- 기획자는 "일단 분석 기능만 만들면 된다"고 생각하고 있습니다.

세 사람이 서로 다른 것을 만든 셈입니다. PRD 한 장이 이 혼란을 막아줍니다.

**비유**: PRD는 건물을 짓기 전에 그리는 설계 도면입니다. 도면 없이 공사를 시작하면 벽이 엉뚱한 곳에 생깁니다.

---

## 2. 핵심 기능을 우선순위로 줄이는 법 — 욕심을 덜어내기

처음 PRD를 쓰면 기능이 10개, 20개가 나옵니다. 16일 안에 다 만들 수 없습니다. **[필수]와 [선택]으로 나누는 것**이 핵심입니다.

**우리 팀이 쓴 기준**

| 질문 | 답이 "예"이면 |
|------|--------------|
| 이 기능 없이는 서비스 자체가 성립하지 않는가? | [필수] |
| 있으면 좋지만 없어도 핵심 흐름은 돌아가는가? | [선택] |
| 16일 남은 일정 안에 만들 수 있는가? | 일정 고려 후 판단 |

**우리 팀 결과**

```
[필수] 3가지
  1. 텍스트 입력 → 위험도 점수
  2. 위험 신호 하이라이트 + 이유
  3. 쉬운 한 줄 경고

[선택] 2가지 (시간 남으면)
  4. 예시 문구 체험 버튼
  5. Pass Rate 표시
```

선택 기능 2개를 V1에서 빼기로 한 이유는 DAY 4 팀 내 의견 조율 결과입니다. 정하린(디자인)은 "UX가 풍부해지니까 넣자"고 했고, 박서준(개발)은 "16일 일정상 필수 3개만 확실히 하자"고 했습니다. PM 이도윤이 "필수 3개를 먼저, 선택 2개는 시간이 남으면"으로 중재했습니다.

**핵심**: 기능을 추가하는 것보다 **줄이는 결정**이 더 어렵고 더 중요합니다.

## 3. 프로토타입으로 "되는지" 빨리 확인하기

프로토타입은 완성된 제품이 아닙니다. **"이 화면이 말이 되는가?"를 빠르게 눈으로 확인하는 스케치**입니다.

DAY 4에서 실제로 작동하는 코드는 하나도 없습니다. 대신 이런 것을 확인했습니다.

1. **화면 배치가 흐름에 맞는가**: 입력창 → 분석 버튼 → 결과(점수·신호·경고) 순서가 자연스러운가?
2. **기술적으로 가능한가**: 규칙 기반(키워드·패턴)으로 점수와 신호를 산출할 수 있는가? → 가능하다고 결론.
3. **빠진 요소가 있는가**: 프로토타입을 보니 "분석 중" 로딩 상태가 필요하다는 걸 발견했습니다.

**비유**: 프로토타입은 요리 전에 그리는 플레이팅 스케치입니다. 실제 요리(코드)는 아직 없지만, "이 그릇에 이렇게 담으면 보기 좋겠다"를 미리 확인합니다.

**도구**: v0(UI 스케치), Cursor Composer(레이아웃 초안). 30분이면 충분합니다. DAY 5 이후에 실제 코드로 구현합니다.

## 4. 기획 / 개발 / 디자인이 처음으로 동시에 움직이는 순간

DAY 1~3는 팀 전체가 같은 작업(문제 정의, 시나리오)을 함께 했습니다. DAY 4부터 처음으로 역할이 나뉩니다.

```
DAY 4 동시 진행

이도윤 (기획/PM)  →  PRD v1 작성
박서준 (개발)     →  화면 스케치 + 기술 검증
정하린 (디자인)   →  화면 흐름 + 색상 기준 초안
최유나 (평가)     →  PRD 성공 지표 ↔ success.md 교차 확인
```

이것이 가능한 이유는 DAY 3에서 이미 **시나리오(흐름)**를 합의했기 때문입니다. "입력창 → 분석 → 결과" 흐름이 정해져 있으니, 기획이 PRD를 완성하는 동안 개발과 디자인이 화면 스케치를 시작할 수 있습니다.

**주의**: 분업은 편리하지만, PRD가 확정되지 않은 상태에서 개발이 코드를 짜기 시작하면 나중에 다시 뜯어야 합니다. DAY 4에서 스케치는 해도 코드 구현은 PRD 확정 후 DAY 5부터 합니다.

## 따라하기: 내 팀 PRD 작성 체크리스트

```
[ ] 한 줄 제품 설명을 한 문장으로 썼다.
    (예: "○○을 하면 ○○을 해주는 도구")

[ ] 핵심 기능을 [필수]와 [선택]으로 나눴다.
    (필수는 없으면 서비스가 안 됨, 선택은 있으면 좋음)

[ ] 사용자 스토리를 "(사용자)는 (목적)을 위해 (행동)을 할 수 있다" 형식으로 썼다.

[ ] 화면 흐름을 3단계(입력 → 처리 → 결과)로 정리했다.

[ ] Out of Scope(이번엔 하지 않는 것)를 명확히 적었다.

[ ] 성공 지표가 success.md의 수치와 일치하는지 확인했다.

[ ] 팀원 모두가 PRD를 읽고 "이거 맞아"라고 했다.
```
