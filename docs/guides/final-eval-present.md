## 1. 최종 Eval로 숫자 확정하기

모든 개선이 끝난 뒤, 마지막으로 한 번 더 평가를 돌립니다. 이것을 최종 Eval이라고 합니다.

**왜 마지막에 한 번 더 돌릴까?**

DAY 8~10에 엔진을 고치고 가드레일을 추가했습니다. 코드가 바뀌면 Pass Rate도 달라질 수 있습니다. "우리 서비스의 최종 성능이 몇 %야?"를 발표 자리에서 정확히 말하려면, 발표 직전에 한 번 더 실측해야 합니다.

> 비유: 시험 공부를 다 마친 뒤 마지막으로 모의고사를 한 번 풀어보는 것과 같습니다. 중간에 공부한 내용이 실제로 반영됐는지 확인하는 과정입니다.

**우리 팀의 최종 Eval**

```
node src/evalRunner.mjs
```

결과: **전체 90.0% (18/20)** — Basic 100%, Edge 83.3%, Safety 83.3%.

이 숫자를 "최종 확정 수치"로 `docs/15-final_report.md`에 기록했습니다.

**잔여 실패 2건도 함께 기록합니다**

90%는 자랑스럽지만, 실패 2건도 발표에 포함합니다.
- case-h01: 키워드 없는 지인 사칭 (미탐)
- case-h02: 드라마 대화 속 '영장' 단어 (오탐)

두 케이스 모두 맥락 이해가 필요합니다. 규칙 기반 엔진의 구조적 한계이고, V3에서 LLM을 보강해야 풀 수 있습니다. 이 한계를 숨기지 않고 발표에 넣는 것이 신뢰를 얻는 길입니다.

## 2. 같은 사용자에게 전/후 묻기 — 왜 DAY 4 테스터가 중요했나

"서비스가 좋아졌다"는 말보다 강력한 증거는 **같은 사람이 전/후에 다르게 반응하는 것**입니다.

**왜 DAY 4에 만난 테스터가 중요한가?**

DAY 4에 인터뷰한 테스터는 서비스가 없던 시절 어떻게 행동했는지를 기억하는 사람입니다. DAY 15에 다시 만나면 "서비스 없이 30초+불안 → 서비스 있이 5초+안심"이라는 비교가 **같은 사람의 입에서** 나옵니다. 이것이 Before/After 비교에서 가장 설득력 있는 증거입니다.

처음 보는 사람에게 서비스를 보여주면 "괜찮은 것 같아요" 정도의 반응이 나옵니다. 하지만 DAY 4에 만나 "의심 문자 받으면 지인에게 묻거나 방치했다"고 했던 테스터가 DAY 15에 "붙여넣고 3~5초 안에 위험 여부와 이유를 확인했다. 왜 위험한지 보여서 안심이 됐다"고 말하면, 그것은 숫자보다 강한 증거입니다.

**Before/After 비교 표 작성법**

| 항목 | Before (서비스 없을 때) | After (서비스 있을 때) |
|------|----------------------|---------------------|
| 소요 시간 | 측정 가능한 숫자로 (예: 30초 이상) | 측정 가능한 숫자로 (예: 3~5초) |
| 판단 근거 | 구체적으로 (예: 감각·지인 의견) | 구체적으로 (예: 점수+신호+이유) |
| 반응 | 테스터가 실제로 한 말 | 테스터가 실제로 한 말 |
| 정확도 | 측정 전 상태 (주관적) | Pass Rate 수치 |

항목은 성공 기준(`docs/03-success.md`)에 적은 "무엇을 측정하는가"와 일치해야 합니다.

## 3. 발표 4대 요소 — Problem·Demo·Eval·Iteration

좋은 발표는 이 네 가지를 순서대로 담습니다.

**Problem (문제)**
- 누구의 어떤 문제인가
- 지금 어떻게 (불편하게) 해결하고 있나
- 왜 AI로 풀기 적합한가

없으면: 발표를 듣는 사람이 "그래서 왜 만든 거야?"를 계속 궁금해합니다.

**Demo (시연)**
- 서비스가 작동하는 모습을 직접 보여줍니다
- 흐름: 사용자가 무엇을 입력하고 → 무엇이 일어나고 → 어떤 결과가 나오는지
- 실행 명령어나 화면 흐름을 구체적으로 제시합니다

없으면: "말로만 좋다고 하는 것"이 됩니다. 직접 보여야 믿습니다.

**Eval (평가)**
- Pass Rate 등 숫자로 성능을 증명합니다
- 개선 전후 비교 (예: 75% → 90%)
- 잔여 한계도 함께 밝힙니다 (숨기면 나중에 더 큰 불신을 낳습니다)

없으면: "좋아진 것 같다"는 주관적 주장에 그칩니다.

**Iteration (반복 개선)**
- V1 → V2 → 가드레일 → 행동 안내로 이어진 개선 흐름
- 피드백을 어떻게 반영했는지
- "한 번에 완성품을 만든 게 아니라, 고치면서 발전했다"는 과정을 보여줍니다

없으면: 결과물만 보이고 과정이 사라집니다. 과정에서 배운 것이 가장 중요한 내용입니다.

## 4. 발표의 주인공은 코드가 아니라 '문제 해결 이야기'

발표에서 기술 설명에 집중하면 듣는 사람이 지칩니다. 가장 기억에 남는 발표는 이야기 구조를 가집니다.

**이야기 구조**

```
[누군가가 불편했다]
  → [우리가 그것을 해결하려 했다]
    → [이런 시도를 했고, 이렇게 실패했으며, 이렇게 고쳤다]
      → [결과: 그 사람이 달라졌다]
```

우리 팀의 발표 이야기 구조:

- 김지우는 의심 문자를 받으면 지인에게 묻거나 방치했습니다. (누군가가 불편했다)
- 우리는 3~5초 안에 위험을 알 수 있는 서비스를 만들기로 했습니다. (해결하려 했다)
- V1은 75%였고, 과적합 경계를 지키며 V2로 90%를 달성했습니다. 가드레일을 넣었고, 피드백으로 행동 안내를 추가했습니다. (시도·실패·개선)
- DAY 15에 김지우는 "왜 위험한지 보여서 안심이 됐다"고 했습니다. (달라졌다)

코드 한 줄보다 이 이야기 한 줄이 더 강력합니다.
