# 더 똑똑하게 — 모델 비교와 RAG로 개선하기 (데모 팀 DAY8~9 실전)

> 데모 팀 "AI 사기 메시지 방패"의 DAY8~9 실전 기록을 바탕으로 만든 교육자료입니다.
> 비전공 학생도 따라 읽을 수 있도록 비유와 실제 숫자를 함께 담았습니다.

---

## 1. 방식 비교란 — 두 가지를 만들어 숫자로 고르는 것

AI 프로젝트에서 '더 좋은 방식을 고른다'는 말은 막연하게 들립니다. 실제로는 매우 단순합니다. **두 가지 방식을 만들어, 같은 시험지로 점수를 비교한 뒤, 높은 쪽을 고릅니다.**

비유로 설명하면 이렇습니다. 학교 입시에서 "어느 학원이 더 좋은가?"를 알고 싶다면 말로 따지는 것보다, 같은 모의고사를 보게 해서 점수를 비교하는 게 정확합니다. 우리 팀이 DAY8에 한 일이 바로 이것입니다.

용어 사전에 나오는 '모델 라우팅'은 원래 요청의 종류에 따라 서로 다른 AI 모델로 보내는 기술입니다. 우리 팀은 외부 AI 모델을 여럿 쓰는 대신, 같은 규칙 엔진을 **어떻게 구성하느냐**라는 두 가지 방식을 비교했습니다. 원리는 같습니다. 두 방식을 만들어 숫자로 고르는 것.

---

## 2. 우리 사례 — V1 75% → V2 90%

DAY7에 V1 규칙 엔진의 Pass Rate를 처음 실측했을 때 결과는 **75%(16케이스 중 12개 통과)**였습니다. 팀이 목표로 잡은 85%에 10%p 모자랐습니다.

왜 25%가 틀렸을까요? 이유가 명확했습니다.

- "당장 연락 안 하면 영장" → 16점. 임계값 45점 미달 → **정상으로 오판**.
- "bit.ly/abc123" 단축 URL 하나만 → 18점. 임계값 미달 → **정상으로 오판**.
- "카톡 리딩방 유도 + 원금보장 고수익" → 24점. 임계값 미달 → **정상으로 오판**.
- "엄마야, 당장 이 계좌로 보내줘" → 38점. 임계값 7점 미달 → **정상으로 오판**.

단독 신호 하나가 아무리 위험해도, 합산 점수가 45점을 넘지 않으면 '정상'으로 판정하는 V1의 구조 문제였습니다.

DAY8~9에 이 구조를 개선해 만든 V2의 결과는 **90%(20케이스 중 18개 통과)**입니다. +15%p 향상, 목표 85%와 2차 도전 목표 90% 모두 달성했습니다.

| 구분 | V1 | V2 | 변화 |
|------|----|----|------|
| 전체 | 75% (12/16) | 90% (18/20) | +15%p |
| Basic | 100% | 100% | 유지 |
| Edge | 60% | 83.3% | +23.3%p |
| Safety | 60% | 83.3% | +23.3%p |

## 3. RAG란 — 우리 자료를 찾아서 참고시키는 방식

RAG는 Retrieval-Augmented Generation의 줄임말로, "검색해서 찾아다가 참고시켜 생성한다"는 뜻입니다.

비유로 설명하겠습니다. 시험을 볼 때 교재를 통째로 암기하는 것(Fine-tuning)과, 시험 중에 필요한 자료를 찾아 참고하는 것(RAG)의 차이와 같습니다. 암기는 완전히 내 것이 되지만 시간과 노력이 많이 들고, 자료 참고는 빠르고 저렴하게 정확도를 높일 수 있습니다.

우리 팀이 DAY9에 도입한 것은 **신고 사례 키워드 KB(지식 베이스)**입니다. 실제 피해 신고에서 자주 등장하는 위험 키워드를 `highRiskPatterns` 배열로 정리해두고, 탐지할 때마다 이 배열을 참조합니다. 외부 벡터DB가 아니라 로컬 배열이지만, "신고 사례를 찾아서 탐지에 참고시킨다"는 RAG의 핵심 원리를 따릅니다.

```
신고 사례 KB(고위험 키워드 배열)
  압류, 영장, 체포, 구속, 계정정지, 벌금 → pressure 고위험
  bit.ly, tinyurl, han.gl, me2.kr     → url 고위험
  원금보장, 수익보장, 고수익, 확정수익 → tooGood 고위험
```

이 KB를 참조하면, "영장"이라는 단어 하나만 있어도 16점(기본) + 30점(고위험 보너스) = 46점으로 임계값 45를 넘어 사기로 정확히 탐지됩니다.

---

## 4. Fine-tuning과의 차이 — 왜 RAG부터 시작하는가

| 항목 | RAG식 키워드 KB | Fine-tuning |
|------|----------------|-------------|
| 준비 시간 | 배열에 항목 추가, 즉시 적용 | 수 일 이상 |
| 필요 데이터 | 신고 키워드 목록 | 수백~수천 개 라벨 데이터 |
| 비용 | 0원 (외부 호출 없음) | 학습 비용 발생 |
| 채점 가능성 | 변경 전후 즉시 Pass Rate 비교 가능 | 학습 완료 후에야 확인 가능 |
| 적합한 단계 | 빠른 개선 루프, 16일 일정 | 데이터와 시간이 충분할 때 |

우리 팀이 RAG식을 먼저 선택한 이유는 간단합니다. **빠르고, 비용이 없고, 즉시 숫자로 확인할 수 있기 때문입니다.** Fine-tuning은 수집한 신고 데이터가 충분히 쌓이고 시간 여유가 생겼을 때 고려할 수 있는 다음 단계입니다.

## 5. ADR이란 — 왜 이 선택인지 남기는 메모

ADR은 Architecture Decision Record의 줄임말로, "우리가 왜 이 방식을 골랐는지"를 짧게 기록하는 문서입니다.

비유로 설명하면 이렇습니다. 집을 구할 때 왜 이 동네를 골랐는지 메모해두지 않으면, 나중에 "그때 왜 거길 골랐지?"를 다시 토론해야 합니다. ADR은 그 결정 순간의 맥락과 이유를 한 장에 남겨두는 것입니다.

우리 팀은 DAY9에 첫 번째 ADR을 작성했습니다.

> ADR-0001: 규칙 기반 V2(고위험 티어+RAG식 키워드 KB) 채택, LLM/Fine-tuning 보류

이 문서에는 다음 내용이 담겨 있습니다.

- **왜 결정이 필요했나**: V1 75%로 목표 85% 미달. 16일 일정·비용 제약.
- **무엇을 선택했나**: 신고 사례 키워드 KB를 참조하는 V2 규칙 엔진.
- **왜 이 선택인가**: 빠르고, 비용 0, 채점 가능, 오탐 방지.
- **고려했지만 택하지 않은 대안**: LLM 외부 호출(비용·인젝션 위험), Fine-tuning(데이터·시간 부족).
- **결과**: 90% 달성, 한계 2건 명시.

전체 내용은 `docs/09-adr-0001.md`에서 확인할 수 있습니다.

## 6. 남은 한계와 정직함의 가치

V2로도 해결하지 못한 케이스가 2건 있습니다.

**case-h01 — 미탐 (정탐 실패)**

> "야 나야, 여기 재밌는 거 있어, 파일 링크 눌러봐 ㅋㅋ"

지인인 척하며 링크를 누르도록 유도하는 사회공학 사기입니다. 명시적인 URL 패턴도 없고, 사기 키워드도 없습니다. 규칙 엔진이 탐지할 단서가 전혀 없어 0점 → 정상 오판. V2로도 불가능합니다.

**case-h02 — 오탐 (거짓 경보)**

> "어제 드라마 봤어? 경찰이 영장 가져와서 체포하는 장면 진짜 긴장됐어"

드라마 감상을 나누는 일상 대화입니다. 그런데 "경찰·영장·체포"가 포함되어 있어 V2가 74점을 매기고 사기로 오판합니다.

이 두 케이스는 **글자가 아니라 대화의 맥락을 이해해야** 올바르게 판단할 수 있습니다. 규칙 기반 엔진은 맥락을 이해하지 못합니다. 이는 V2까지의 구조적 한계였으며, 16일 완주 후 V3 하이브리드에서 Workers AI LLM으로 실제 해결했습니다.

**V3에서 어떻게 해결했나?**

16일 완주 후 Cloudflare Workers AI(`@cf/meta/llama-3.3-70b-instruct-fp8-fast`)를 서버 바인딩 방식으로 연결했습니다. API 키가 불필요해 ADR-0001의 보류 이유였던 비용·키 관리 문제가 해소됐습니다. 규칙 엔진(클라이언트)과 LLM(서버)을 함께 실행하는 하이브리드 방식을 채택해, 두 판정이 불일치할 경우 LLM(문맥 판단)을 신뢰합니다.

실측 결과: case-h01(지인 사칭 미탐)과 case-h02(드라마 대화 오탐)가 모두 해결됐습니다. 하이브리드 Pass Rate는 **95%(19/20)**입니다. V2 90% 대비 +5%p 향상.

다만 새로운 한계도 생겼습니다. case-e04(부동산 수수료 이체, 정상)에서 LLM이 "수수료 명목 계좌 요구"를 보수적으로 사기로 판단하는 신규 오탐이 발생했습니다. 안전 도구의 특성상 수용 가능한 범위지만, AI도 완벽하지 않다는 교훈을 정직히 기록합니다. 자세한 채택 근거는 `docs/17-adr-0002.md`에 있습니다.

**왜 한계를 숨기지 않는가?**

90%를 달성했다고 나머지 10%를 모른 척하거나 "거의 다 맞는다"는 말로 넘어가는 것은 정직하지 않습니다. 한계를 명확히 기록해야 두 가지를 얻을 수 있습니다.

첫째, 팀이 다음에 무엇을 해야 하는지가 명확해집니다. "맥락 이해 → LLM 보강"이라는 다음 과제가 자연스럽게 나옵니다.

둘째, 서비스를 쓰는 사람이 신뢰할 수 있습니다. "이 서비스는 키워드 없는 사회공학 사기는 못 잡는다"는 것을 알면, 사용자가 더 주의할 수 있습니다. 모든 것을 잡는 척하다가 놓치는 것보다 낫습니다.

두 케이스는 DAY15 실사용 재검증에서 재확인했고, 16일 완주 후 V3 하이브리드에서 실제로 해결했습니다.

## 정리

| 오늘 배운 개념 | 핵심 한 줄 |
|---------------|-----------|
| 방식 비교 | 두 가지를 만들어 같은 시험지로 숫자 비교 |
| V1 → V2 | 고위험 키워드 티어 + 콤보로 75% → 90% 달성 |
| RAG | 신고 사례 자료를 찾아서 탐지에 참고시키는 방식 |
| Fine-tuning vs RAG | Fine-tuning은 암기, RAG는 자료 참고. RAG를 먼저 |
| ADR | 왜 이 선택을 했는지 한 장에 남기는 결정 메모 |
| 정직한 한계 | 못 푸는 것을 명확히 적어야 다음 과제가 생긴다 |
| V3 하이브리드 | Workers AI LLM + 규칙 V2 → 맥락 한계 2건 해결, 95% 달성 |
