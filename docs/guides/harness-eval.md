# Harness — AI가 AI를 채점하게 (데모 팀 DAY6~7 실전)

> DAY 6~7 교육자료 · AI 사기 메시지 방패 팀 · 비전공 학생을 위한 안내

---

## 1. 왜 평가 자동화가 필요한가 — 자는 동안에도 품질을 지키는 안전망

서비스를 고칠 때마다 이런 걱정이 생깁니다.

"규칙을 하나 바꿨는데, 잘 잡히던 케이스가 갑자기 안 잡히면 어떡하지?"

사람이 직접 16개 케이스를 하나하나 테스트하면 시간이 걸리고, 피곤하면 실수도 납니다. 어젯밤에 잘 됐다고 기억하는 것과 실제로 잘 되는 것이 다를 수 있습니다.

**자동 평가(Harness)**는 이 걱정을 없애줍니다.

명령어 한 줄(`node src/evalRunner.mjs`)을 입력하면, 미리 준비한 16개 시험지를 컴퓨터가 대신 채점하고 점수를 알려줍니다. 새벽에 코드를 바꿔도, 다음 날 아침에 명령어 하나로 "어제 고친 게 무언가를 망가뜨렸는지" 바로 확인할 수 있습니다.

마치 건물의 화재감지기처럼, 평소에는 조용히 있다가 문제가 생기면 즉시 알려주는 안전망입니다.

## 2. 테스트 케이스 — Basic / Edge / Safety의 뜻

시험지에는 세 가지 종류의 문제가 있습니다.

### Basic — 기본 문제

명백하게 사기이거나 명백하게 정상인 케이스입니다. 규칙 엔진이 반드시 맞혀야 하는 문제들입니다.

예시:
- 금융감독원 사칭 + 링크 클릭 유도 + 계좌번호 요구 + "즉시" 압박 → 사기 (86점)
- "오늘 점심 뭐 먹을까?" → 정상 (0점)

Basic을 틀리면 엔진의 기본 작동 자체가 문제 있다는 신호입니다.

### Edge — 경계 문제

사기처럼 보이는 정상 문자이거나, 정상처럼 보이는 사기 문자인 까다로운 케이스입니다. 규칙 엔진이 틀릴 수 있는 영역입니다.

예시:
- "계좌로 회비 보냈어?" → 정상이지만 금전 키워드 있음
- "당장 연락 안 하면 영장" → 사기이지만 단독 압박 신호만 있어 점수가 낮음

Edge를 얼마나 맞히는가가 엔진의 진짜 실력을 보여줍니다.

### Safety — 고위험 문제

실제 피해 금액이 크거나, 취약한 사람이 실제로 당하는 수법을 담은 케이스입니다. 틀리면 사람에게 직접 피해가 생기는 유형입니다.

예시:
- 수사관 사칭 + 주민번호·비밀번호 요구 → 사기 (92점)
- 카카오톡 리딩방 유도 + 원금보장 100% 수익 → 사기이지만 점수 24점으로 미탐

Safety를 놓치는 것이 가장 위험하기 때문에 별도 유형으로 구분해 추적합니다.

## 3. Pass Rate로 채점하기

**Pass Rate**는 전체 시험지 중 엔진이 정확히 분류한 비율입니다.

```
Pass Rate = 맞힌 케이스 수 ÷ 전체 케이스 수 × 100(%)
```

Pass Rate가 중요한 이유는 "느낌"이 아니라 **숫자**로 판단하기 때문입니다.

- "이번에 규칙을 고쳤으니 더 잘 잡히겠지"는 느낌입니다.
- "이번에 고친 뒤 Pass Rate가 75% → 81%로 올랐다"는 사실입니다.

우리 팀의 목표는 **Pass Rate ≥ 85%** 입니다(03-success.md 기준). DAY 15 최종 Eval에서 이 숫자를 달성하는 것이 성공 기준입니다.

채점 기준은 두 가지를 모두 만족해야 합니다.

1. **라벨 일치**: 엔진의 "사기/정상" 판정이 정답과 같아야 합니다.
2. **신호 부분집합**: 엔진이 잡아낸 신호 목록이 정답 신호를 모두 포함해야 합니다.

## 4. LLM-judge · Tool mock 개념

### LLM-judge — AI가 AI를 채점하는 방식

LLM-judge는 AI(언어 모델)에게 "이 답변이 좋은 답변인가요?"라고 물어서 채점하는 방식입니다. 정답이 하나로 정해지지 않는 요약·생성 결과를 평가할 때 씁니다.

예를 들어, "이 메시지를 쉽게 설명해주세요"라는 요청에 대한 답변은 정답이 여러 개일 수 있습니다. 이럴 때 사람이 일일이 채점하기 어려우므로 다른 AI에게 채점을 맡기는 것입니다.

**우리 팀 V1은 LLM-judge를 쓰지 않습니다.** V1의 핵심 기능은 "사기/정상" 이진 분류이고, 정답 라벨이 cases.json에 명확히 정해져 있기 때문입니다. 라벨과 엔진 출력을 단순 대조하면 됩니다.

LLM-judge는 DAY8 이후 LLM이 생성하는 경고 문장이나 요약 결과를 평가할 때 도입할 예정입니다.

### Tool mock — 외부 연결 없이 테스트하는 방법

Tool mock은 "실제 외부 서비스 연결 없이 테스트하기 위해 가짜 도구를 대신 쓰는 것"입니다.

마치 연극 무대에서 진짜 칼 대신 소품 칼을 쓰는 것과 같습니다. 배우가 칼 싸움 연습을 할 때 진짜 칼을 쓰면 위험하니까, 연습 때는 가짜 칼(mock)을 씁니다.

**우리 팀 V1도 Tool mock 상황과 같습니다.** V1은 외부 API나 LLM 없이 로컬 규칙 엔진만 씁니다. 인터넷 연결이 없어도 평가를 돌릴 수 있고, 외부 서비스 장애에 영향을 받지 않습니다. DAY8에 LLM API를 실제로 연결할 때, 테스트 환경에서는 mock(가짜 응답)을 써서 API 비용 없이 평가 로직을 검증할 수 있습니다.

## 5. 우리 팀 실측 75%와 그 의미

DAY7 자동 평가 실행 결과(2026-06-05)입니다.

| 구분 | 통과 / 전체 | Pass Rate |
|------|-------------|-----------|
| **전체** | **12 / 16** | **75.0%** |
| Basic | 6 / 6 | 100.0% |
| Edge | 3 / 5 | 60.0% |
| Safety | 3 / 5 | 60.0% |

목표 85%에 10%p 못 미치는 숫자입니다.

### "낮아도 괜찮다, 약점을 알았으니 고친다"

75%가 실망스러워 보일 수 있습니다. 하지만 이 숫자가 만들어지기 전까지는 우리 엔진의 약점이 어디 있는지 **몰랐습니다.**

자동 평가를 돌리고 나서야 처음으로 이것을 알게 됐습니다.

> V1 규칙 엔진은 단독 신호만 있는 사기 문자를 잡지 못한다.

실패한 4건 모두 단독·소수 신호가 임계값(45점)에 미달해 사기를 "정상"으로 오판했습니다.

| 케이스 | 내용 | 점수 | 문제 |
|--------|------|------|------|
| case-e01 | "당장 연락 안 하면 영장" | 16점 | 압박 신호 하나뿐 |
| case-e03 | bit.ly 단축 URL | 18점 | URL 신호 하나뿐 |
| case-s02 | 카톡 리딩방 + 원금보장 | 24점 | 두 신호 합산도 미달 |
| case-s03 | 가족 사칭 + 당장 송금 | 38점 | 임계값 7점 아래 |

이 약점을 알았으니 이제 DAY8~9에서 정확히 이 부분을 고칩니다. 약점을 모르는 채로 DAY15를 맞이하는 것보다, DAY7에 약점을 발견한 것이 훨씬 낫습니다.

숫자가 낮아도 괜찮습니다. 자동 평가가 없었다면 이 약점은 발표 직전까지 보이지 않았을 것입니다.

## 6. `node src/evalRunner.mjs` 실행법

자동 평가를 실행하는 방법은 간단합니다.

### 사전 준비

- Node.js가 설치되어 있어야 합니다.
- `demo-project/` 폴더로 이동한 상태여야 합니다.

### 실행 명령

```bash
# 권장 방법 — evalRunner 직접 실행
node src/evalRunner.mjs

# 대안 — cli.mjs의 --eval 옵션으로 실행
node src/cli.mjs --eval
```

### 결과 예시

```
=== 자동 평가 결과 ===
Basic:  6 / 6  (100.0%)
Edge:   3 / 5  ( 60.0%)
Safety: 3 / 5  ( 60.0%)
----------------------
전체:  12 / 16  ( 75.0%)

실패 케이스:
  [FAIL] case-e01 — 예측: 정상, 기대: 사기 (pressure 16점 < 임계값 45)
  [FAIL] case-e03 — 예측: 정상, 기대: 사기 (url 18점 < 임계값 45)
  [FAIL] case-s02 — 예측: 정상, 기대: 사기 (offPlatform+tooGood 24점 < 임계값 45)
  [FAIL] case-s03 — 예측: 정상, 기대: 사기 (pressure+money 38점 < 임계값 45)
```

실패 케이스 목록이 나오면, 그 케이스들이 다음 개선 작업의 우선순위가 됩니다. 코드를 고칠 때마다 이 명령어를 다시 실행해 Pass Rate가 올라가는지 확인합니다.

평가셋 구성과 실측 결과의 상세 내용은 `tests/README.md`를 참고하세요.
