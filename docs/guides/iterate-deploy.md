## 1. 피드백을 '반영 항목'으로 바꾸기

데모를 마치면 피드백이 쌓입니다. 그런데 피드백을 그냥 읽어두면 아무것도 바뀌지 않습니다. 반드시 '무엇을 언제 고친다'는 항목으로 변환해야 합니다.

**우리 팀(DAY 12 → DAY 13)의 예시**

| 피드백 | 반영 항목 | 시점 |
|--------|---------|------|
| "그래서 어떻게 해야 하나가 안 나와요" | 행동 안내 한 줄 추가 (`actionAdvice`) | DAY 13 필수 |
| "드라마 대화인데 위험하다고 나왔어요" | 오탐 한계 안내 문구 추가 | DAY 13 필수 |
| "결과가 색으로 확 구분됐으면" | 결과 색 강조 UI | 향후 차기 |
| "예시 버튼이 있으면 좋겠어요" | 예시 문구 체험 버튼 | 향후 차기 |

피드백 4건 중 이번 16일 안에 핵심 2건만 반영하고, 나머지 2건은 '차기 백로그'로 기록했습니다. 이것이 **범위 관리**(scope management)입니다. 모든 요청을 지금 당장 다 반영하려다가 아무것도 제대로 못 만드는 것보다, 핵심을 골라 확실히 반영하는 편이 낫습니다.

**정리 공식**: `피드백 → (중요도 분류) → 반영 항목 + 담당자 + 시점`

## 2. Changelog란 — 무엇이 왜 바뀌었나

Changelog(체인지로그)는 버전과 버전 사이에 **무엇이 추가·변경·수정됐는지, 왜 그랬는지**를 기록하는 문서입니다.

> 비유: 약 봉투 뒷면의 "이전 제품 대비 개선점"과 같습니다. 새 버전을 받은 사람이 "뭐가 달라진 거야?"를 보는 곳입니다.

**Changelog 작성의 세 가지 원칙**

1. **Added·Changed·Fixed로 나누기**: 새로 생긴 것, 바뀐 것, 고쳐진 것을 구분하면 읽기 쉽습니다.
2. **왜 바뀌었는지를 한 줄로**: "DAY 12 피드백 #1 반영", "DAY 7 평가 실패 4건이 원인" 처럼 근거를 적습니다. 근거 없는 변경은 나중에 "왜 이렇게 됐지?"를 알 수 없어 되돌리기가 어렵습니다.
3. **커밋 기록 활용하기**: 실제 코드 변경은 Git 커밋 메시지에 이미 기록되어 있습니다. `git log --oneline`으로 커밋 목록을 보고, AI에 "이 커밋들로 Changelog 작성해줘"라고 요청하면 초안을 빠르게 만들 수 있습니다.

**우리 팀의 Changelog 예시** (`docs/13-changelog.md`)

```
## [v2] - DAY 13
### Added
- 행동 안내(actionAdvice) — DAY 12 피드백 #1 반영
- 고위험 티어·RAG식 KB — DAY 8 V2 구현

### Changed
- Pass Rate 75% → 90%

### Fixed
- 단독 강신호 미탐 4건 해소

### 변경 근거
- DAY 12 피드백 #1/#2, DAY 7 평가 실패 4건 분석
```

## 3. 버전 표기 — PRD v2, TRD v4

문서에도 버전이 있습니다. 처음 만든 것은 v1, 한 번 크게 고치면 v2입니다.

**왜 버전을 표기할까?**

- 팀원이 "오늘 기준 최신 스펙이 뭐야?"를 문서만 보고 알 수 있습니다.
- "이 기능, v1 때부터 있던 거야, v2에 새로 생긴 거야?"를 추적할 수 있습니다.
- 발표 자료나 보고서에서 "DAY 4 기획서(v1) 대비 실제 구현(v2 기준)"을 비교할 수 있습니다.

**우리 팀의 버전 이력**

| 문서 | v1 | v2 이후 |
|------|-----|---------|
| PRD (`04-prd.md`) | DAY 4 최초 작성 | DAY 13 v2 — 행동 안내·한계 안내 추가 |
| TRD (`05-trd.md`) | DAY 5 최초 작성 | DAY 13 v4 — V2 엔진·가드레일 4종·실측 90% 반영 |

TRD가 v4인 이유: DAY 8(V2 엔진), DAY 10(가드레일), DAY 13(행동 안내 필드·실측 기록)에 한 번씩 더 갱신했기 때문입니다.

## 4. 정식 배포 = 처음 보는 사람도 5분 내 실행 가능하게

배포(deploy)라는 단어가 어렵게 느껴질 수 있습니다. 핵심만 기억하세요.

> **정식 배포 = 처음 보는 사람이 README만 읽고 5분 안에 직접 실행할 수 있는 상태**

그 전까지는 '내 PC에서만 돌아가는 것'입니다.

**README가 배포의 핵심인 이유**

우리 저장소는 `node src/cli.mjs "문자 내용"`으로 즉시 실행됩니다. 하지만 이것을 저장소를 처음 본 사람이 알 수 있으려면 README에 명확히 써야 합니다. README가 없거나 불친절하면, 아무리 잘 만든 서비스도 아무도 쓰지 못합니다.

**좋은 README의 4요소**

1. 이게 무엇인지(한 줄 소개)
2. 무슨 문제를 해결하는지(해결하는 문제)
3. 어떻게 실행하는지(실행 방법 — 복사해서 바로 붙여넣을 수 있는 명령어)
4. 누가 만들었는지(팀)

## 5. 커스텀 도메인·HTTPS 개념 — 자물쇠의 의미

브라우저 주소창 왼쪽에 있는 자물쇠 아이콘을 본 적 있으시죠? 이것이 HTTPS입니다.

> 비유: HTTP는 엽서입니다. 내용이 그대로 보입니다. HTTPS는 봉투에 넣어 잠근 편지입니다. 중간에 가로채도 내용을 볼 수 없습니다.

- **HTTP**: 데이터가 인터넷을 이동할 때 암호화되지 않습니다. 공공 와이파이에서 사용하면 내용이 노출될 수 있습니다.
- **HTTPS**: 데이터를 암호화해서 전송합니다. 브라우저가 자물쇠로 표시합니다.
- **커스텀 도메인**: `github.io/scam-shield` 대신 `scam-shield.com` 같은 자체 주소를 쓰는 것입니다. 기억하기 쉽고 신뢰감을 줍니다.

우리 팀은 Cloudflare를 통해 이 둘을 구성할 계획입니다. Cloudflare는 도메인을 사고, DNS(주소록)를 관리하고, HTTPS 인증서를 무료로 발급해 주는 서비스입니다.

## 6. 우리 사례 — DAY 13~14 실전 요약

**Pass Rate 75% → 90%**: DAY 7에 측정한 기준선 75%가 V2 구현(고위험 티어·RAG식 KB·콤보 보너스) 후 90%로 올랐습니다. 숫자로 개선을 증명한 것이 핵심입니다.

**행동 안내 추가**: "위험하다"는 것을 알려주는 것에서 끝나지 않고, "그래서 어떻게 해야 하나"까지 답합니다. 피드백 한 마디가 서비스를 한 단계 발전시켰습니다.

**Changelog 링크**: `docs/13-changelog.md`에 v2와 v1의 차이를 Added/Changed/Fixed/변경 근거로 정리했습니다. 발표 때 "무엇을 왜 바꿨는지"를 이 파일 한 장으로 설명할 수 있습니다.

**README 정식화**: DAY 14에 README에서 "DAY0 초안" 경고를 제거하고, 처음 보는 사람도 5분 내 실행할 수 있도록 실행 명령어·기술 스택·팀을 정비했습니다.
