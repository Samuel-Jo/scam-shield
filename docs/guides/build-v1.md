# 작동하는 V1 먼저 — 부끄러워도 내놓기 (데모 팀 DAY5 실전)

> DAY 5 교육자료 · AI 사기 메시지 방패 팀 · 비전공 학생을 위한 안내

---

## 1. 왜 완벽보다 '작동하는 최소'가 먼저인가

새 서비스를 만들 때 가장 흔한 실수는 "조금만 더 다듬고 보여주자"를 반복하다가 아무것도 내놓지 못하는 것입니다. 코드에는 항상 고칠 것이 있어 보이고, 화면은 언제나 부족해 보입니다.

하지만 실제로 필요한 것은 **'지금 작동하는 최소 버전'** 입니다.

- 완벽하지 않아도 됩니다. 틀린 부분이 있어도 됩니다.
- 중요한 것은 실제로 실행해보고 "무엇이 문제인지"를 빨리 발견하는 것입니다.
- 머릿속 계획이 아무리 완벽해도, 실제로 돌려보기 전까지는 어디가 틀렸는지 모릅니다.

이것을 **최소 기능 제품(Minimum Viable Product, MVP)** 이라고 부릅니다. 자동차를 만든다면, 처음부터 고급 세단을 만들려 하기보다 우선 굴러가는 킥보드를 먼저 만드는 방식입니다.

---

## 2. 조기 배포의 힘 — 일찍 보여줘야 일찍 고친다

우리 팀의 목표는 DAY 15 최종 발표에서 **Pass Rate ≥ 85%** 를 달성하는 것입니다. 이 숫자를 달성하려면 테스트를 하고, 틀린 케이스를 찾고, 규칙을 수정하는 '개선 루프'를 여러 번 돌려야 합니다.

개선 루프는 V1을 내놓기 전까지는 시작조차 할 수 없습니다.

```
[V1 내놓기] → [테스트로 약점 발견] → [규칙 수정] → [다시 테스트] → ...
```

DAY 5에 V1을 내놓으면 DAY 7 테스트셋으로 약점을 찾고, DAY 8에 LLM을 도입해 개선하고, DAY 12 중간 데모에서 검증할 수 있습니다. 반대로 V1을 DAY 8까지 미루면 그만큼 개선할 기회가 줄어듭니다.

**부끄러운 버전을 일찍 내놓는 것 자체가 전략입니다.**

---

## 3. 규칙 기반으로 시작한 이유 — 빠르고 채점이 쉽다

우리 팀의 V1은 LLM(대형 언어 모델)을 쓰지 않습니다. 대신 "외부 링크 유도", "개인정보 요청"처럼 사기 문자에 자주 등장하는 7종 패턴을 직접 정의한 **규칙 기반** 방식으로 작동합니다.

규칙 기반을 먼저 선택한 이유는 두 가지입니다.

**첫째, 빠릅니다.** LLM API를 연동하고 프롬프트를 설계하는 데는 며칠이 필요합니다. 규칙 기반은 패턴 목록만 정의하면 당일에 작동하는 버전을 만들 수 있습니다.

**둘째, 채점이 쉽습니다.** 규칙이 명확하면 "왜 이 결과가 나왔는지"를 바로 추적할 수 있습니다. 어떤 규칙이 틀렸는지 파악한 뒤 고치면 Pass Rate가 올라갑니다. LLM은 강력하지만, 왜 틀렸는지 파악하기가 더 어렵습니다.

> 규칙 기반은 Pass Rate 85% 달성의 빠른 출발점입니다. 한계에 부딪히면 DAY 8에 LLM으로 강화합니다.

## 4. CLI로 먼저 만든 이유

CLI(Command Line Interface)는 터미널 창에 명령어를 입력해 프로그램을 실행하는 방식입니다. 화면(웹 UI)이 없고, 텍스트만 나옵니다.

처음 보면 낯설고 불편해 보이지만, 개발 초기에는 오히려 CLI가 유리합니다.

- **화면을 만드는 시간을 아낄 수 있습니다.** 규칙 엔진이 제대로 작동하는지 확인하는 것이 먼저입니다. 화면은 나중에 붙여도 됩니다.
- **입력과 결과를 바로 확인할 수 있습니다.** 텍스트를 입력하면 점수와 신호 목록이 즉시 터미널에 나옵니다.
- **오류 메시지가 바로 보입니다.** 화면이 없으면 오히려 어디서 문제가 생겼는지 찾기 쉽습니다.

우리 팀은 CLI로 먼저 규칙 엔진의 작동을 확인하고, DAY 12 중간 데모 때 웹 화면을 추가하기로 합의했습니다.

## 5. 우리 팀 V1 실제 — detectScam과 CLI 실행

박서준이 DAY 5에 구현한 핵심 함수는 `detectScam`입니다.

```js
// src/scamDetector.mjs 일부 (개념 예시)
export function detectScam(text) {
  // 7종 신호를 순서대로 탐지하고 점수를 누적합니다
  const signals = [];
  let score = 0;

  if (/링크|클릭|접속/.test(text)) {
    signals.push("외부 링크 클릭 유도");
    score += 20;
  }
  if (/개인정보|계좌번호|비밀번호/.test(text)) {
    signals.push("개인정보 수집 시도");
    score += 25;
  }
  // ... (나머지 5종 신호)

  return {
    score: Math.min(score, 100),
    signals,
    warning: score >= 70
      ? "사기 가능성이 높습니다. 링크를 누르거나 개인정보를 입력하지 마세요."
      : score >= 40
      ? "일부 위험 신호가 감지됐습니다. 주의하세요."
      : "위험 신호가 적습니다. 그래도 출처를 확인하세요.",
  };
}
```

터미널에서 실행하는 방법은 다음과 같습니다.

```bash
node src/cli.mjs
```

실행하면 "분석할 문자를 붙여넣으세요:"라는 안내가 나오고, 텍스트를 입력하면 결과가 출력됩니다.

```
--- 분석 결과 ---
위험도 점수: 75 / 100  [위험]
감지된 신호:
  - 외부 링크 클릭 유도
  - 개인정보 수집 시도
  - 긴급성 조성

경고: 사기 가능성이 높습니다. 링크를 누르거나 개인정보를 입력하지 마세요.

* 이 서비스는 참고용입니다. 최종 판단은 사용자 본인이 합니다.
```

전체 코드 구조와 실행 방법의 최신 내용은 `demo-project/src/README.md`를 참고하세요.

## 6. V1의 한계와 다음 단계

V1 규칙 기반에는 명확한 한계가 있습니다.

| 한계 | 이유 |
|------|------|
| 새로운 사기 수법은 탐지 못할 수 있음 | 미리 정의한 패턴에 없는 문구는 놓칩니다 |
| "아래 링크"가 모두 사기는 아님 | 정상 안내 문자도 링크를 포함하는 경우가 있어 오탐(false positive)이 생길 수 있습니다 |
| 문맥 이해 불가 | "이 링크는 클릭하지 마세요"처럼 부정 문맥도 위험으로 잘못 분류할 수 있습니다 |

이 한계를 해결하기 위해 다음 단계가 예정되어 있습니다.

- **DAY 7**: 테스트셋(15개 이상) 구축 → Pass Rate 첫 측정, 틀린 케이스 목록 작성
- **DAY 8**: LLM 도입 → 문맥 이해 능력 추가, V1 규칙이 놓친 케이스 보완
- **DAY 9**: RAG(검색 증강 생성) 도입 → 최신 사기 수법 사례를 참고 자료로 활용

V1은 완성품이 아닙니다. DAY 15까지 계속 개선되는 **출발점**입니다.
