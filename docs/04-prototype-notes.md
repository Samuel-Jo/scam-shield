## 무엇으로 만들었나

v0(Vercel의 UI 프로토타이핑 도구)와 Cursor Composer를 활용해 약 30분 동안 정적 화면 스케치를 만들었습니다. 클릭 이벤트나 실제 분석 로직은 없으며, 입력창·버튼·결과 영역의 배치와 정보 구조를 눈으로 확인하는 것이 목적입니다. 실제 엔진(키워드 분류·점수 산출 코드)은 DAY 5부터 구현합니다.

## 기술 실현 가능성 점검

**질문: 규칙 기반 V1(키워드·패턴 매칭)으로 위험도 점수와 신호 산출이 가능한가?**

검토 결과: 가능합니다.

| 기능 | V1 구현 방식 (규칙 기반) | 판단 |
|------|--------------------------|------|
| 위험도 점수 산출 | 위험 키워드/패턴 목록을 미리 정의하고, 입력 텍스트에서 매칭된 수와 가중치를 합산해 0~100점으로 환산 | 가능 |
| 위험 신호 하이라이트 | 매칭된 키워드/구문의 위치(인덱스)를 반환해 프론트에서 강조 표시 | 가능 |
| 이유 설명 (한 줄) | 패턴 종류별로 미리 작성해 둔 설명 문자열을 매칭 결과에 연결해 출력 | 가능 |
| 쉬운 한 줄 경고 | 위험도 구간(높음/보통/낮음)별로 미리 작성한 경고 문장 3종을 선택해 출력 | 가능 |

규칙 기반 방식의 한계(새 수법·변형 문구 대응 어려움)는 DAY 5 이후 LLM 통합으로 보완할 예정입니다. V1은 핵심 흐름(입력→분석→결과)이 작동하는지 검증하는 데 집중합니다.

**결론**: DAY 5 구현을 진행합니다.

## DAY 4 분업 안착 메모

DAY 4는 팀에서 처음으로 역할이 나뉘어 동시에 움직인 날입니다.

| 역할 | 담당자 | DAY 4 작업 |
|------|--------|------------|
| 기획 (PM) | 이도윤 | PRD v1 작성 — 핵심 기능 5개 우선순위 확정, 사용자 스토리·Out of Scope 정리 |
| 개발 | 박서준 | 정적 화면 스케치 제작, 규칙 기반 V1 기술 실현 가능성 검토 |
| 디자인 | 정하린 | 화면 3요소(입력창·분석버튼·결과영역) 흐름 정의, 위험도 색상 기준 초안 제안 |
| 평가 | 최유나 | PRD 성공 지표 항목이 03-success.md와 일치하는지 교차 확인 |

- 기획(PRD)이 완성되기 전에 개발·디자인이 화면 스케치를 시작할 수 있었던 이유: 03-scenario.md에서 이미 "입력창 → 분석 → 결과(점수·신호·경고)" 흐름이 합의되어 있었기 때문입니다.
- PRD가 확정되는 순간, 개발은 DAY 5 구현 범위를 즉시 특정할 수 있는 상태가 되었습니다.
