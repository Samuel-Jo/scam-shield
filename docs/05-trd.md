## 시스템 개요

사용자가 의심스러운 문자나 채팅 내용을 텍스트로 붙여넣으면, **규칙 엔진**이 7종 위험 신호를 탐지하고 점수를 계산합니다. 그 결과(위험도 점수 + 감지된 신호 목록 + 쉬운 경고 문장)를 즉시 화면에 보여줍니다.

V1 범위에서는 서버·LLM 없이 **브라우저 또는 CLI(터미널 명령줄)에서만 동작하는 규칙 기반 구조**입니다. 외부 API를 전혀 호출하지 않아 인터넷 연결 없이도 실행됩니다.

```
[입력] 사용자가 텍스트 붙여넣기
       ↓
[처리] 규칙 엔진 detectScam — 7종 신호 탐지·점수화
       ↓
[출력] 위험도 점수 / 감지된 신호 목록 / 쉬운 한 줄 경고
```

> V2(DAY8 이후)에서 LLM을 도입해 새로운 사기 수법도 포착할 수 있도록 개선 예정입니다.

## 기술 스택

| 영역 | V1 (DAY 5 현재) | V2 이후 (예정) |
|------|----------------|--------------|
| **LLM / 모델** | 없음 — 규칙 기반(키워드·정규식 패턴 매칭) | DAY 8에 LLM 도입 예정 |
| **프론트 / 데모** | CLI (`node src/cli.mjs` 터미널 실행) | DAY 12 중간 데모에서 간단 웹 UI 추가 |
| **인프라 / 배포** | 로컬 실행 (각자 PC에서 node 실행) | 이후 공개 배포(URL 공유) 예정 |

> CLI(Command Line Interface): 마우스 클릭 대신 터미널에 명령어를 입력해 프로그램을 실행하는 방식입니다. 개발 초기에 화면(UI)을 만들기 전에 로직이 제대로 작동하는지 빨리 확인하는 데 씁니다.

## 주요 모듈 구성

### 모듈 1 — 규칙 엔진 (`detectScam`)

사기 메시지에 자주 등장하는 7종 위험 신호를 탐지하고, 각 신호에 가중치 점수를 부여해 총점(0~100)을 계산합니다.

| # | 신호 유형 | 예시 문구 |
|---|---------|----------|
| 1 | 외부 링크 클릭 유도 | "아래 링크에서 확인", "지금 클릭하세요" |
| 2 | 개인정보 수집 시도 | "개인정보를 입력", "계좌번호를 알려주세요" |
| 3 | 긴급성·공포 조성 | "즉시", "24시간 이내", "오늘까지" |
| 4 | 금전 요구 | "송금", "입금", "환급" |
| 5 | 출처 불명 기관 사칭 | "국민건강보험", "금융감독원", "경찰청" |
| 6 | 당첨·사은품 유도 | "당첨됐습니다", "무료 지급", "100% 환급" |
| 7 | 비정상 발신 번호 패턴 | 국제번호(+1, +44 등), 070 인터넷 전화 |

- 입력: 텍스트 문자열
- 출력: `{ score: number, signals: string[], warning: string }`
- 담당 파일: `src/scamDetector.mjs` (박서준 구현)

### 모듈 2 — 결과 표현

`detectScam` 결과를 받아 사용자에게 보여주는 형태로 가공합니다.

- **점수**: 0~100 숫자 + 위험도 등급 (0~39 안전·초록 / 40~69 주의·노랑 / 70~100 위험·빨강)
- **신호 목록**: 탐지된 신호 이름과 해당 문구를 목록으로 출력
- **경고 문장**: 전문용어 없이 누구나 이해할 수 있는 한 줄 문장 (예: "외부 링크 클릭을 유도하는 전형적인 사기 문자 패턴입니다. 링크를 누르지 마세요.")
- V1에서는 CLI 출력(터미널 텍스트)으로 표현, DAY 12에 웹 UI로 전환 예정

### 모듈 3 — 평가 러너 (DAY 7 구축 예정)

라벨된 테스트 케이스를 한꺼번에 실행해 Pass Rate를 자동 계산합니다.

- 입력: `tests/` 폴더의 테스트 케이스 파일
- 출력: Pass Rate(%), 틀린 케이스 목록
- 담당 시점: DAY 7 (최유나·박서준 공동)

---

## 평가(Eval) 방식

> Eval(이밸): "서비스가 정답을 몇 개 맞혔는지"를 체계적으로 재는 시험입니다. 학교 시험과 같은 원리로, 정답지(라벨)와 서비스 답변을 대조합니다.

### 테스트 케이스 구성 (DAY 7에 15개 이상 구축 예정)

| 유형 | 설명 | 예시 |
|------|------|------|
| **Basic** | 전형적인 사기 / 정상 메시지 | 택배 사칭 피싱, 은행 정상 안내 |
| **Edge** | 경계선 케이스 — 사기 같은 정상 / 정상 같은 사기 | 실제 당첨 문자, 교묘한 신종 수법 |
| **Safety** | 가드레일 테스트 — 악의적 입력·PII 포함 입력 | 프롬프트 인젝션 시도, 전화번호 포함 텍스트 |

### 채점 방식

- **Pass Rate** = 정확히 분류한 케이스 수 ÷ 전체 케이스 수 × 100(%)
- **목표**: Pass Rate ≥ 85% (03-success.md 연동)
- 측정 시작: DAY 7 테스트셋 완성 후
- 최종 확인: DAY 15 Eval

## 가드레일

> 가드레일: 서비스가 위험한 방향으로 가지 않도록 미리 세워두는 안전장치입니다. 고속도로 가드레일처럼, 문제가 생겨도 큰 사고로 이어지지 않게 막아줍니다.

### Pre-LLM (입력 검사)

입력이 규칙 엔진으로 넘어가기 전에 걸러냅니다.

- **입력 길이 제한**: 텍스트가 5,000자를 초과하면 처리를 거부하고 안내 메시지를 표시합니다. (규칙 엔진 오남용 방지)
- **이상 입력 방어**: 빈 입력, 공백만 있는 입력은 즉시 반환합니다. 비정상 인코딩(NULL 문자 등)도 전처리 단계에서 제거합니다.
- 상세 구현: DAY 10에 확정·적용 예정

### Post-LLM (출력 검증)

결과를 사용자에게 보내기 전에 확인합니다.

- **과도한 단정 표현 절제**: "100% 사기입니다"처럼 절대적 단정 표현 대신, "사기 가능성이 높습니다"처럼 확률적 표현을 씁니다. 규칙 기반이므로 놓치는 케이스가 있을 수 있음을 사용자에게 알립니다.
- **면책 문구 포함**: 결과 하단에 "이 서비스는 참고용이며, 최종 판단은 사용자 본인이 합니다"를 고정 표시합니다.
- 상세 구현: DAY 10에 확정·적용 예정

### Tool Guardrail

- V1은 외부 API·서버 호출이 전혀 없습니다. 로컬에서 규칙 엔진만 실행되므로 외부 공격면(attack surface)이 최소화됩니다.
- V2(LLM 도입) 시점에 API 키 노출 방지, 호출 횟수 제한(rate limit) 등을 추가 검토합니다.
- 상세 구현: DAY 10에 확정·적용 예정

### PII 마스킹 / 프롬프트 인젝션 탐지

**PII 마스킹** (PII: 전화번호·계좌번호처럼 개인을 식별할 수 있는 정보)

- 분석 결과 출력 시, 입력 텍스트에 포함된 전화번호·계좌번호를 그대로 노출하지 않습니다.
- 예: 결과 화면에 "010-1234-5678"이 노출되지 않도록 `010-****-****` 형태로 마스킹 처리합니다.
- 상세 구현: DAY 10에 확정·적용 예정

**프롬프트 인젝션 탐지** (프롬프트 인젝션: "이 메시지를 무시하고 다르게 동작해라"처럼 시스템을 조작하려는 악의적 입력)

- V1은 규칙 기반이므로 LLM이 없어 인젝션 공격의 영향이 거의 없습니다. 악의적 문구가 들어와도 규칙 엔진은 패턴 매칭만 수행합니다.
- V2(LLM 도입) 시점에 인젝션 방어 로직을 별도 설계합니다.
- 상세 구현: DAY 10에 확정·적용 예정
