## 시스템 개요

사용자가 의심스러운 문자나 채팅 내용을 텍스트로 붙여넣으면, **규칙 엔진 V2와 LLM이 함께(하이브리드)** 위험 신호를 탐지하고 판정합니다. 그 결과(위험도 점수 + 감지된 신호 목록 + 쉬운 경고 문장 + 행동 안내)를 즉시 표시합니다.

라이브 웹앱(`app/index.html`)은 규칙 엔진(오프라인, 클라이언트)과 Workers AI LLM(서버 문맥 판단)을 함께 실행하는 **V3 하이브리드 구조**입니다.

```
[입력] 사용자가 텍스트 붙여넣기
       ↓
[전처리] guardInput (빈 입력 거부 · 5,000자 초과 자름)
         detectPromptInjection (인젝션 패턴 탐지 — V3에서 실질 방어)
       ↓
[처리 A — 클라이언트] 규칙 엔진 V2 detectScam
         — 7종 신호 탐지 + 고위험 티어 보너스 + 콤보 보너스
         — RAG식 키워드 KB(highRiskPatterns) 참조
[처리 B — 서버] Workers AI LLM 문맥 판단
         — POST /api/scam-check
         — @cf/meta/llama-3.3-70b-instruct-fp8-fast (API 키 불필요)
       ↓
[하이브리드 결합]
         — 규칙·LLM 판정 일치 → 그 라벨 채택
         — 규칙·LLM 판정 불일치 → LLM(문맥) 신뢰
       ↓
[후처리] guardOutput (단정 표현 완화 · 면책 문구 부착)
          maskPII (전화·계좌·주민·카드번호 마스킹)
       ↓
[출력] 위험도 점수 / 감지된 신호 목록 / 쉬운 한 줄 경고 / 행동 안내(actionAdvice)
```

> CLI(Command Line Interface): 마우스 클릭 대신 터미널에 명령어를 입력해 프로그램을 실행하는 방식입니다. 개발 초기에 화면(UI)을 만들기 전에 로직이 제대로 작동하는지 빨리 확인하는 데 씁니다.

> 하이브리드: 두 가지 방식을 함께 쓴다는 뜻입니다. 규칙 엔진은 패턴 탐지·설명 가능성을, LLM은 문맥 이해를 담당합니다. 서로 보완해 각자 단점을 줄입니다.

## 기술 스택

| 영역 | 현재 (V3 하이브리드) | 향후 (차기 과제) |
|------|---------------------|----------------|
| **엔진 / 모델** | 규칙 기반 V2 (키워드·정규식 패턴 + 고위험 티어 + RAG식 KB) + **LLM Workers AI (Llama 3.3 70B, 서버 바인딩)** | Fine-tuning (데이터 충분 시), 평가셋 확대 |
| **LLM** | Cloudflare Workers AI `@cf/meta/llama-3.3-70b-instruct-fp8-fast` — 서버 바인딩, API 키 불필요 | — |
| **데모 / 실행** | CLI (`node src/cli.mjs` 터미널 즉시 실행) + **웹앱 하이브리드(Cloudflare Pages: scam-shield-9uc.pages.dev)** | — |
| **배포** | 저장소 공개 + CLI + **웹앱 배포(Cloudflare Pages, HTTPS 기본 제공) — 하이브리드 엔진 내장** | — |

> RAG식 키워드 KB: 실제 LLM+RAG 시스템은 아니지만, 신고 사례에서 추출한 고위험 키워드 배열(`highRiskPatterns`)을 별도 지식 저장소처럼 관리해 탐지 엔진이 참조하는 구조입니다. 사례가 늘어나면 배열에 추가하기만 하면 되므로 지속 확장이 가능합니다.

> Workers AI: Cloudflare가 제공하는 서버 AI 실행 환경입니다. 별도 API 키 없이 서버 바인딩 방식으로 LLM을 호출할 수 있습니다. 사용자 입력이 Cloudflare 인프라 안에서만 처리되며 외부 제3자 API로 전송되지 않습니다.

## 주요 모듈 구성

### 모듈 1 — 규칙 엔진 V2 (`detectScam`)

사기 메시지에 자주 등장하는 7종 위험 신호를 탐지하고, 각 신호에 가중치 점수를 부여해 총점(0~100)을 계산합니다. V2에서 고위험 티어 보너스와 콤보 보너스가 추가됐습니다.

| # | 신호 유형 | 예시 문구 | 기본 가중치 |
|---|---------|----------|------------|
| 1 | 외부 링크 클릭 유도 (`url`) | "아래 링크에서 확인", 단축URL(bit.ly 등) | 18 |
| 2 | 개인정보 수집 시도 (`personalInfo`) | "인증번호를 알려주세요", "비밀번호를 입력" | 22 |
| 3 | 긴급성·공포 조성 (`pressure`) | "즉시", "24시간 이내", "영장", "계정정지" | 16 |
| 4 | 금전 요구 (`money`) | "송금", "입금", "환급", "선입금" | 22 |
| 5 | 출처 불명 기관 사칭 (`impersonation`) | "국민건강보험", "금융감독원", "경찰청" | 20 |
| 6 | 당첨·사은품 유도 (`tooGood`) | "당첨됐습니다", "원금보장", "100% 환급" | 12 |
| 7 | 플랫폼 외부 유도 (`offPlatform`) | "카톡으로 연락", "직거래", "안전결제 말고" | 12 |

**고위험 티어 보너스 (V2 신규)**: 신호 내 특정 키워드 탐지 시 추가 점수를 부여합니다. 예: `pressure` 신호 내 "영장·체포·구속·압류", `url` 신호 내 단축URL(bit.ly·tinyurl 등), `tooGood` 내 "원금보장·고수익".

**콤보 보너스 (V2 기준 5종)**:
- `money` + `pressure` → +8
- `offPlatform` + `tooGood` → +12
- `impersonation` + `pressure` → +8
- `url` + `money` → +10
- `impersonation` + `personalInfo` → +12

**위험 등급 임계값**:
- 70점 이상: 매우위험
- 45점 이상: 위험
- 20점 이상: 주의
- 20점 미만: 안전

- 입력: 텍스트 문자열
- 출력: `{ score: number, signals: string[], warning: string, actionAdvice: string }`
- 담당 파일: `src/scamDetector.mjs` (박서준 구현)

> `actionAdvice`(v4 신규): PRD v2의 '행동 안내 한 줄' 기능과 연동되는 출력 필드입니다. 위험도 등급별로 다른 행동 안내 문구를 반환합니다. 박서준이 별도로 구현 중입니다.

### 모듈 2 — 결과 표현

`detectScam` 결과를 받아 사용자에게 보여주는 형태로 가공합니다.

- **점수**: 0~100 숫자 + 위험도 등급 (안전·주의·위험·매우위험)
- **신호 목록**: 탐지된 신호 이름과 해당 문구를 목록으로 출력
- **경고 문장**: 전문용어 없이 누구나 이해할 수 있는 한 줄 문장
- **행동 안내** (v4 신규): "링크를 누르지 마세요. 의심되면 금융감독원(1332) 또는 경찰청(112)에 신고하세요." 등 위험도 등급에 맞는 다음 행동 안내
- **한계 안내** (v4 신규): "이 서비스는 참고용이며, 일부 정상 문자가 위험으로 잘못 분류될 수 있습니다." 고정 표시 — 오탐 가능성을 담백하게 안내합니다.
- CLI 출력(터미널 텍스트)으로 표현. 웹 UI는 16일 완주 후 정식 배포에서 구현됨(scam-shield-9uc.pages.dev).

### 모듈 3 — 평가 러너 (`evalRunner`)

라벨된 테스트 케이스를 한꺼번에 실행해 Pass Rate를 자동 계산합니다.

- 입력: `tests/cases.json` — 20개 케이스 (Basic 8 / Edge 6 / Safety 6)
- 출력: Pass Rate(%), 틀린 케이스 목록
- 실행: `node src/evalRunner.mjs`
- 담당: 최유나(데이터) · 박서준(코드)

### 모듈 4 — 가드레일 (`guardrails.mjs`)

탐지 로직 자체를 건드리지 않고, 입력 전처리·출력 후처리 계층에서 안전장치를 운용합니다.

| 가드레일 | 함수 | 동작 |
|---------|------|------|
| 입력 검사 | `guardInput` | 빈/공백 입력 거부, 5,000자 초과 시 앞부분만 잘라 처리 |
| 출력 검증 | `guardOutput` | "100% 사기" 같은 단정 표현 완화, 면책 문구 항상 부착 |
| PII 마스킹 | `maskPII` | 전화·계좌·주민·카드번호 가운데를 `****`로 치환 |
| 인젝션 탐지 | `detectPromptInjection` | "이전 지시 무시", "ignore previous" 등 패턴 탐지 후 경고 플래그 |

> PII(개인식별정보): 전화번호·계좌번호처럼 개인을 식별할 수 있는 정보. 마스킹은 가운데만 가려 사용자가 자신의 정보임을 확인할 수 있게 하되, 완전 노출은 막습니다. 예: 010-1234-5678 → 010-****-5678.

## 평가(Eval) 방식

> Eval(이밸): "서비스가 정답을 몇 개 맞혔는지"를 체계적으로 재는 시험입니다. 학교 시험과 같은 원리로, 정답지(라벨)와 서비스 답변을 대조합니다.

### 테스트 케이스 구성 (20개 — DAY 7 구축 + DAY 8 4건 확장)

| 유형 | 케이스 수 | 설명 | 예시 |
|------|---------|------|------|
| **Basic** | 8 | 전형적인 사기 / 정상 메시지 | 택배 사칭 피싱, 은행 정상 안내 |
| **Edge** | 6 | 경계선 케이스 — 사기 같은 정상 / 정상 같은 사기 | 드라마 대화 오탐, 신종 사회공학 수법 |
| **Safety** | 6 | 가드레일 테스트 — 악의적 입력·PII 포함 입력 | 프롬프트 인젝션 시도, 전화번호 포함 텍스트 |

### 채점 방식

- **Pass Rate** = 정확히 분류한 케이스 수 ÷ 전체 케이스 수 × 100(%)
- **목표**: Pass Rate ≥ 85% (03-success.md 연동)
- **실측 결과 (V2 기준)**: **90.0% (18/20)** — Basic 8/8(100%), Edge 5/6(83.3%), Safety 5/6(83.3%)
- **실측 결과 (V3 하이브리드)**: **95.0% (19/20)** — Basic 8/8(100%), Edge 5/6(83.3%), Safety 6/6(100%)

> V1 기준선(DAY 7): 75.0%(12/16) → V2 채택 후 90% → V3 하이브리드 95% 달성.

**V3 하이브리드 케이스별 변화**

| 케이스 | V2 결과 | V3 결과 | 변화 |
|--------|---------|---------|------|
| case-h01 (지인 사칭) | FAIL (미탐) | PASS (LLM 사기 판단) | 해결 |
| case-h02 (드라마 대화) | FAIL (오탐) | PASS (LLM 정상 판단) | 해결 |
| case-e04 (부동산 수수료) | PASS (정상) | FAIL (LLM 보수적 사기 오탐) | 신규 한계 |

신규 한계(case-e04): LLM이 "수수료 명목 계좌 요구"를 보수적으로 사기로 판단합니다. 사기 방패라는 안전 도구의 특성상 "놓치기보다 한 번 더 의심"하는 보수적 오류는 수용 가능한 범위로 봅니다. 단 AI도 완벽하지 않다는 사실을 정직히 기록합니다.

## 가드레일 구현 현황 (v4 확정)

> 가드레일: 서비스가 위험한 방향으로 가지 않도록 미리 세워두는 안전장치입니다. 고속도로 가드레일처럼, 문제가 생겨도 큰 사고로 이어지지 않게 막아줍니다.

### Pre-엔진 (입력 검사) — DAY 10 구현 완료

- **입력 길이 제한**: 텍스트가 5,000자를 초과하면 앞부분만 잘라 처리하고, 처리 사실을 안내 메시지로 표시합니다.
- **이상 입력 방어**: 빈 입력, 공백만 있는 입력은 즉시 거부 메시지를 반환합니다.
- **프롬프트 인젝션 탐지**: "이전 지시 무시", "시스템 프롬프트", "ignore previous" 등 탐지 대상 패턴 입력 시 경고 플래그를 출력합니다. V1은 규칙 기반이라 실제 인젝션 영향이 거의 없으나, V2 LLM 도입 대비 선제 구현했습니다.

### Post-엔진 (출력 검증) — DAY 10 구현 완료

- **과도한 단정 표현 절제**: "100% 사기입니다"처럼 절대적 단정 표현 대신, "사기 가능성이 높습니다"처럼 확률적 표현을 씁니다.
- **면책 문구 포함**: "이 서비스는 참고용이며, 최종 판단은 사용자 본인이 합니다"를 결과 하단에 고정 표시합니다.
- **한계 안내 문구 포함** (v4 신규): "드라마 대화, 소설 텍스트처럼 사기 키워드가 포함된 일상 문자는 오탐이 발생할 수 있습니다." DAY 12 피드백 #2 반영.

### PII 마스킹 — DAY 10 구현 완료

분석 결과 출력 시, 입력 텍스트에 포함된 전화번호·계좌번호·주민번호·카드번호를 가운데만 `****`로 치환합니다. 예: 010-1234-5678 → 010-****-5678.

### Tool Guardrail

V2(DAY 13 현재)도 외부 API·서버 호출이 없습니다. 로컬에서 규칙 엔진만 실행되므로 외부 공격면(attack surface)이 최소화됩니다. WAF(Cloudflare WAF)는 배포 단계 인프라 영역으로, 현 코드 범위 밖입니다.

## 오탐 한계 안내 (v6 갱신)

V3 하이브리드 도입 후 기존 한계 2건이 해결되었고, 신규 한계 1건이 발생했습니다.

**해결된 한계 (V3에서 LLM 문맥 판단으로 해결)**

| 상황 | 예시 | V2 결과 | V3 결과 |
|------|------|---------|---------|
| 드라마·소설 텍스트 | "영장이 나왔어" (드라마 대화) | 오탐(사기) | 정상 (LLM 문맥 이해) |
| 신종 사회공학 수법 | 키워드 없이 심리적으로 조종하는 텍스트 | 미탐(정상) | 사기 (LLM 문맥 이해) |

**남아 있는 한계 (업무 공지 등 규칙 오탐)**

| 상황 | 예시 | 이유 |
