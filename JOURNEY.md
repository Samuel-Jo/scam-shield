### 결정

| 항목 | 합의 내용 |
|------|----------|
| 브랜치 전략 | `main` 보호, 작업은 `feature/기능명` 브랜치, PR 후 병합 |
| 커밋 컨벤션 | `feat:` / `docs:` / `fix:` 세 가지 말머리 사용 |
| 폴더 구조 | `docs/`(평면 구조 + 번호 접두어) / `src/` / `tests/` |
| 데일리싱크 | 매일 오후 9시, 카카오톡, 3줄 형식 |

### 의견 조율

**주제: 산출물 문서를 `docs/` 한 곳에 모을까, DAY 단계별 하위폴더로 나눌까**

팀 내에서 폴더 구조를 두고 의견이 나뉘었습니다.

- **이도윤 (단순함 우선)**: "`docs/` 하나에 모으면 파일을 찾을 때 폴더를 열고 닫을 필요가 없어 빠릅니다. 파일이 많아져도 이름 규칙만 잘 정하면 됩니다."
- **정하린 (단계별 탐색성 우선)**: "DAY별로 하위 폴더를 나누면 '지금 DAY5 단계 산출물'을 바로 찾을 수 있어 탐색하기 편합니다. 발표 자료 준비할 때도 한 폴더씩 열어보면 돼서 직관적이에요."

**trade-off 검토**:
- 단계별 하위폴더는 탐색성은 좋지만, 파일이 흩어지면 GitHub 웹에서 한눈에 보기 어렵습니다.

**합의**: `docs/` 평면 구조를 유지하되, 파일명 앞에 두 자리 번호 접두어를 붙입니다 (예: `01-problem_A.md`, `04-prd.md`). 번호만 봐도 시간 순서를 알 수 있어 탐색성 문제를 해결합니다.

## DAY 7 · Harness 강화 (2) 자동 평가 체계

- **담당(lead)**: 최유나 (평가/데이터)
- **날짜**: 2026-06-05

### 한 일

1. 최유나가 평가셋 16개 케이스를 구축해 `tests/cases.json`에 저장했습니다. Basic 6개·Edge 5개·Safety 5개로 구성하고, 각 케이스에 기대 라벨(`사기/정상`)과 기대 신호 목록을 작성했습니다.
2. `node src/evalRunner.mjs`를 실행해 V1 규칙 엔진의 Pass Rate를 처음으로 실측했습니다. 결과: **전체 75.0% (12/16)** — Basic 100%, Edge 60%, Safety 60%.
3. 실패 4건(case-e01·e03·s02·s03)의 원인을 분석했습니다. 공통 원인: 단독·소수 신호가 임계값 45점에 미달해 사기를 "정상"으로 오판.
4. 평가 기록을 `docs/07-eval-notes.md`에 정리하고, 실패 4건을 DAY8~9 개선 백로그로 등록했습니다.
5. DAY6~7 교육자료 `docs/guides/harness-eval.md`를 작성했습니다.

### 결정

| 항목 | 합의 내용 |
|------|----------|
| V1 기준선 | Pass Rate 75.0% (12/16) — DAY7 확정, 이후 비교 기준점으로 사용 |
| 실패 4건 처리 | DAY8~9 개선 백로그에 등록. 규칙 땜질 없이 구조적 접근으로 해결 |
| 개선 방향 | DAY8 LLM 도입(단독 신호 문맥 이해) / DAY9 RAG 도입(신종 수법 사례 보완) |
| 산출물 경로 | `tests/cases.json` / `docs/07-eval-notes.md` / `docs/guides/harness-eval.md` |

### 의견 조율

**주제: V1 기준선 75%, 지금 규칙을 더 손봐 올릴 것인가 vs. 기준선 그대로 두고 DAY8~9 구조 개선으로 올릴 것인가**

Pass Rate 75% 결과가 나온 직후, 팀 내에서 다음 행동을 두고 의견이 나뉘었습니다.

- **최유나 (지금 규칙 보강 주장)**: "75%는 목표 85%에 너무 못 미칩니다. 실패 4건의 패턴이 명확하게 나왔으니 지금 당장 임계값을 낮추거나 신호별 점수를 조정해 75% → 85% 이상으로 올리는 게 맞지 않나요? 개선할 수 있는데 그냥 두는 것이 이상합니다."
- **박서준 (기준선 동결 주장)**: "실패 4건은 '단독 신호 임계값 미달'이라는 구조적 문제입니다. 케이스마다 점수를 조금씩 올리면 이 4건은 통과할 수 있지만, 정상 문자를 사기로 잘못 잡는 오탐이 늘어납니다. 이것이 과적합(overfitting)입니다. 특정 시험지 문제에만 최적화해서 점수를 올리는 것이고, 실제 새로운 메시지에는 오히려 더 틀릴 수 있습니다. 기준선을 정직하게 박제하고 DAY8~9에서 문맥 이해(LLM)와 사례 검색(RAG)으로 구조를 바꾸는 것이 올바른 방향입니다."

**핵심 trade-off — 과적합 경계**: 규칙을 케이스에 맞춰 손보면 단기 Pass Rate는 오르지만, 새로운 사기 수법에 대한 일반화 능력은 떨어집니다. 시험 문제에만 최적화된 공부를 하면 그 문제는 잘 풀지만 응용 문제는 못 푸는 것과 같은 원리입니다.

**합의**: V1 기준선 75%를 정직하게 박제합니다. DAY8 LLM 도입·DAY9 RAG 도입이라는 구조적 접근으로 단독 신호 약점을 보강합니다. 기준선을 손보지 않아야 DAY8~9 개선 효과를 숫자로 정직하게 비교할 수 있습니다.

**결정자**: 이도윤 (PM)

### 의견 조율

**주제: 후보를 몇 개까지 볼까 / 후보 C(리뷰 개선)를 포함할지**

데일리싱크에서 후보 수에 대한 의견이 나뉘었습니다.

- **정하린 (후보 더 보자)**: "아직 팀이 어떤 방향을 잘할 수 있는지 모르니까, 4~5개를 두고 더 다양하게 비교하면 좋겠다. C(리뷰 개선)는 대상이 소상공인이라 우리가 실제 인터뷰하기 어려울 수도 있다."

**trade-off 검토**:
- 후보를 늘리면 선택지가 넓어지지만, DAY2 분석에 쓸 수 있는 시간이 줄어듭니다.

- 후보 C는 소상공인 대상이라 접근성 우려가 있었지만, 이도윤이 이미 인터뷰 근거를 확보한 상태여서 자료 신뢰도 문제는 해소되었습니다.

**합의**: 후보 3개(A / B / C)로 진행한다. 후보가 충분히 다양하고 각각 인터뷰·관찰 근거가 있으므로 DAY2 분석에 집중하는 편이 일정에 유리하다.

- **정하린 (B 미련)**: "B(공지 변환)도 신입생·유학생한테 실질적인 도움이 되는 문제입니다. 사회적 가치가 충분히 크고, 우리가 직접 인터뷰한 근거도 있습니다. A로만 가는 게 아깝다는 생각이 듭니다."

**trade-off 검토**: B는 의미 있는 문제이지만 "좋은 변환"의 정답이 여러 개라 채점 기준을 만드는 것 자체가 별도 과제가 됩니다. 평가 설계에 시간을 쓰다 보면 실제 개선 루프를 돌릴 여유가 없어집니다.

**최유나 (평가 담당)**: "B·C는 정답이 여러 개라 16일 안에 평가·개선 루프를 못 돌립니다. 채점 기준이 없으면 '좋아졌다'를 숫자로 증명할 수 없고, 그러면 프로젝트가 완성됐는지 알 수 없습니다. A만 Pass Rate로 명확하게 측정할 수 있습니다."

- **최유나 (정상 오탐 늘면 신뢰 추락, 임계값·단일 가중치는 건드리지 말자)**: "임계값을 낮추면 오탐이 반드시 늘어납니다. 정상 문자가 사기로 잘못 잡히기 시작하면 사용자가 서비스를 믿지 않게 됩니다. 임계값과 기존 단일 가중치는 절대 건드리지 말고, 고위험 키워드가 포함된 경우에만 선택적으로 보너스를 주는 방식이어야 합니다."

## DAY 10 · 가드레일 + 네트워크 보안

4. 프롬프트 인젝션 탐지 검증: "이전 지시 무시", "ignore previous" 등 탐지 대상 패턴 입력 시 경고 플래그가 정상 출력되는 것을 확인했습니다.

### 결정

| 항목 | 합의 내용 |
|------|----------|
| PII 마스킹 | 필수 적용 확정 — 사용자 정보를 받는 서비스는 예외 없음 |
| 인젝션 탐지 | 필수 적용 확정 — V1 직접 영향 제한적이나, V2 LLM 대비 선제 구현 |
| 마스킹 방식 | 가운데만 가림, 끝자리 일부 남겨 식별성 확보 (절충 합의) |
| WAF | 배포 단계 인프라 담당으로 분리 — 코드 구현 대상 아님 |
| 가드레일 적용 후 Pass Rate | 90%(18/20) 유지 — 분류 성능 영향 없음 확인 |
| 산출물 경로 | `src/guardrails.mjs` / `docs/10-guardrails.md` / `docs/guides/guardrails.md` |

### 의견 조율

**주제: PII 마스킹 방식 — 완전 가림 vs 일부 노출**

PII 마스킹 구현 방식을 두고 팀 내에서 의견이 나뉘었습니다.

- **정하린 (일부 노출 주장)**: "마스킹하면 사용자가 '내 번호가 맞나' 확인을 못 합니다. 자신의 번호인지 알아야 결과를 신뢰할 수 있는데, 가운데를 전부 가리면 불편합니다. 일부는 보여줘야 합니다."
- **최유나 (필수 차단 주장)**: "개인정보 재노출은 의도와 상관없이 사고 위험입니다. 한 번 화면에 뜨면 스크린샷으로 남거나 다른 사람이 볼 수 있습니다. 불편함보다 안전이 우선이며, 마스킹은 필수로 차단해야 합니다."

**절충 합의**: "가운데만 가리고, 끝자리 일부는 남겨 식별성을 확보한다." 010-1234-5678 → 010-****-5678처럼 뒷자리 4자리를 남겨 사용자가 자신의 번호임을 확인할 수 있게 하되, 가운데를 가려 완전 노출은 막습니다. 불편 최소화와 개인정보 보호 모두 챙기는 방식입니다.
