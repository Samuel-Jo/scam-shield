### 결정

| 항목 | 합의 내용 |
|------|----------|
| 브랜치 전략 | `main` 보호, 작업은 `feature/기능명` 브랜치, PR 후 병합 |
| 커밋 컨벤션 | `feat:` / `docs:` / `fix:` 세 가지 말머리 사용 |
| 폴더 구조 | `docs/`(평면 구조 + 번호 접두어) / `src/` / `tests/` |
| 데일리싱크 | 매일 오후 9시, 카카오톡, 3줄 형식 |

### 의견 조율

**주제: 산출물 문서를 `docs/` 한 곳에 모을까, DAY 단계별 하위폴더로 나눌까**

팀 내에서 폴더 구조를 두고 의견이 나뉘었습니다.

- **이도윤 (단순함 우선)**: "`docs/` 하나에 모으면 파일을 찾을 때 폴더를 열고 닫을 필요가 없어 빠릅니다. 파일이 많아져도 이름 규칙만 잘 정하면 됩니다."
- **정하린 (단계별 탐색성 우선)**: "DAY별로 하위 폴더를 나누면 '지금 DAY5 단계 산출물'을 바로 찾을 수 있어 탐색하기 편합니다. 발표 자료 준비할 때도 한 폴더씩 열어보면 돼서 직관적이에요."

**trade-off 검토**:
- 단계별 하위폴더는 탐색성은 좋지만, 파일이 흩어지면 GitHub 웹에서 한눈에 보기 어렵습니다.
- 평면 구조(`docs/` 한 곳)는 단순하지만, 파일이 많아지면 정렬 기준이 중요해집니다.

**합의**: `docs/` 평면 구조를 유지하되, 파일명 앞에 두 자리 번호 접두어를 붙입니다 (예: `01-problem_A.md`, `04-prd.md`). 번호만 봐도 시간 순서를 알 수 있어 탐색성 문제를 해결합니다.

## DAY 1 · 문제 후보 도출

- **담당(lead)**: 정하린 (UX) · 이도윤 (PM) 공동
- **날짜**: 2026-06-04

### 한 일

2. 박서준이 중고거래 커뮤니티와 주변 지인 경험을 바탕으로 **후보 A(사기 메시지 탐지)**를 정리했습니다.
3. 정하린이 신입생 오픈채팅방 관찰과 유학생 인터뷰를 바탕으로 **후보 B(학교 공지 쉬운말 변환)**를 정리했습니다.
4. 이도윤이 동네 자영업자 인터뷰와 커뮤니티 게시글 관찰을 바탕으로 **후보 C(리뷰 → 사장님 개선 리포트)**를 정리했습니다.
5. 페인 포인트는 "○○를 했더니 △△해서 불편" 형태의 구체적인 장면으로 기록했습니다. 추상적인 감상이 아니라 관찰·인터뷰 근거를 담았습니다.
6. `docs/guides/problem-discovery.md`에 비전공 학생이 참고할 수 있는 DAY1 교육자료를 작성했습니다.

### 결정

| 항목 | 합의 내용 |
|------|----------|
| 후보 수 | A / B / C 3개로 확정 |
| 최종 선택 시점 | DAY2에서 AI 적합성 필터를 적용해 1개 선택 (DAY1에는 선택하지 않음) |
| 산출물 경로 | `docs/01-problem_A.md` / `01-problem_B.md` / `01-problem_C.md` |
| 교육자료 경로 | `docs/guides/problem-discovery.md` |

### 의견 조율

**주제: 후보를 몇 개까지 볼까 / 후보 C(리뷰 개선)를 포함할지**

데일리싱크에서 후보 수에 대한 의견이 나뉘었습니다.

- **정하린 (후보 더 보자)**: "아직 팀이 어떤 방향을 잘할 수 있는지 모르니까, 4~5개를 두고 더 다양하게 비교하면 좋겠다. C(리뷰 개선)는 대상이 소상공인이라 우리가 실제 인터뷰하기 어려울 수도 있다."
- **박서준 (3개면 충분, 분석에 집중)**: "후보가 늘어나면 DAY2 분석 시간이 길어진다. 3개를 제대로 분석하는 게 낫다. C는 이도윤이 실제 인터뷰까지 한 상태니까 근거가 있다."

**trade-off 검토**:
- 후보를 늘리면 선택지가 넓어지지만, DAY2 분석에 쓸 수 있는 시간이 줄어듭니다.
- 3개로 제한하면 비교가 빠르고, AI 적합성 필터를 더 꼼꼼히 적용할 수 있습니다.
- 후보 C는 소상공인 대상이라 접근성 우려가 있었지만, 이도윤이 이미 인터뷰 근거를 확보한 상태여서 자료 신뢰도 문제는 해소되었습니다.

**합의**: 후보 3개(A / B / C)로 진행한다. 후보가 충분히 다양하고 각각 인터뷰·관찰 근거가 있으므로 DAY2 분석에 집중하는 편이 일정에 유리하다.

### 결정

| 항목 | 합의 내용 |
|------|----------|
| 최종 선택 문제 | 후보 A — 사기 메시지 탐지 |
| 확정 문제 한 문장 | "사기 문자·중고거래 채팅 메시지를 받은 사용자가 사기인지 정상인지 스스로 판단하지 못해 피해를 입거나 불필요한 불안을 겪는다." |
| 한 줄 비전 | "사기 문자를 받은 순간, 누구나 3초 안에 위험을 알아챈다." |
| 범위 (다룬다) | 텍스트 분류 + 위험신호 추출 + 쉬운 경고 출력 |
| 범위 (다루지 않는다) | 이미지 판독, 통화 음성, 실시간 자동 차단, 신고 자동 접수 |
| 정하린의 B 아이디어 | '확장 후보'로 기록. 16일 프로젝트 완주 후 다음 기회에 검토. |
| 산출물 경로 | `docs/02-problem.md` / `docs/02-vision.md` / `docs/guides/ai-fit-filter.md` |

### 의견 조율

**주제: 후보 A(사기 탐지) vs 후보 B(공지 변환) — 최종 선택**

AI 적합성 필터 결과를 발표한 뒤, B에 대한 미련이 남아있던 정하린이 의견을 냈습니다.

- **박서준 (A 지지)**: "A는 '사기/정상' 두 라벨로 채점 기준이 명확합니다. 테스트 케이스 만들고 Pass Rate 숫자로 보여주면 되니까 평가 설계가 단순합니다. 16일 안에 완주할 수 있는 구조입니다."
- **정하린 (B 미련)**: "B(공지 변환)도 신입생·유학생한테 실질적인 도움이 되는 문제입니다. 사회적 가치가 충분히 크고, 우리가 직접 인터뷰한 근거도 있습니다. A로만 가는 게 아깝다는 생각이 듭니다."

**trade-off 검토**: B는 의미 있는 문제이지만 "좋은 변환"의 정답이 여러 개라 채점 기준을 만드는 것 자체가 별도 과제가 됩니다. 평가 설계에 시간을 쓰다 보면 실제 개선 루프를 돌릴 여유가 없어집니다.

**최유나 (평가 담당)**: "B·C는 정답이 여러 개라 16일 안에 평가·개선 루프를 못 돌립니다. 채점 기준이 없으면 '좋아졌다'를 숫자로 증명할 수 없고, 그러면 프로젝트가 완성됐는지 알 수 없습니다. A만 Pass Rate로 명확하게 측정할 수 있습니다."

이 채점 가능성 근거가 결정적이었습니다. 팀 전체가 A로 수렴했습니다. 정하린의 B 아이디어는 즉시 폐기하지 않고 '확장 후보'로 기록해 두어, 16일 완주 후 다음 프로젝트에서 검토할 여지를 남겼습니다.

## DAY 3 · 사용자 시나리오 + 성공 기준

- **담당(lead)**: 정하린 (UX) · 이도윤 (PM)
- **날짜**: 2026-06-05

### 한 일

1. 정하린이 사용자 시나리오 `docs/03-scenario.md`를 작성했습니다. 대표 사용자 김지우(24세, 대학생)의 Before → 사용 → After 흐름을 구체적인 장면으로 서술하고, 결정적 순간("위험도 점수와 위험 이유가 동시에 보이는 순간")을 명확히 정의했습니다.
2. 이도윤이 MVP 성공 기준 `docs/03-success.md`를 작성했습니다. "기존 30초에서 5초 안에 판단 + Pass Rate ≥ 85%"라는 측정 가능한 한 문장으로 성공 기준을 확정했습니다.
3. 측정 방법을 구체화했습니다. Pass Rate는 라벨된 테스트셋(최소 16개) 대조로, 판단 소요 시간은 초기 테스터의 전/후 스톱워치 측정으로 재는 방식을 합의했습니다.
4. DAY3 교육자료 `docs/guides/success-criteria.md`를 작성했습니다. 비전공 학생이 '측정 가능한 성공 기준'의 3요소(무엇을·어떻게·목표 수치)와 결정적 순간 찾는 법을 실제 팀 산출물을 인용해 따라할 수 있도록 구성했습니다.

### 결정

| 항목 | 합의 내용 |
|------|----------|
| 성공 기준 (1차) | Pass Rate ≥ 85% AND 판단 시간 ≤ 5초 |
| 성공 기준 (2차 도전) | V2(LLM 개선) 단계에서 Pass Rate ≥ 90% 도전 |
| 판단 시점 | DAY 15 최종 Eval + 초기 테스터 재검증 |
| 결정적 순간 | 문자 붙여넣기 직후 위험도 점수와 위험 이유가 동시에 표시되는 순간 |
| 산출물 경로 | `docs/03-scenario.md` / `docs/03-success.md` / `docs/guides/success-criteria.md` |

### 의견 조율

**주제: Pass Rate 목표를 90%로 높게 잡을 것인가, 85%로 현실적으로 잡을 것인가**

성공 기준 수치를 두고 팀 내에서 의견이 나뉘었습니다.

- **최유나 (높은 목표 지지)**: "어차피 목표면 높게 잡아야 팀이 더 노력하지 않나요? Pass Rate 90%로 목표를 잡아야 의미가 있다고 생각합니다. 85%는 너무 낮은 것 같아요."
- **박서준 (현실 기반 목표 지지)**: "V1은 규칙 기반(키워드·패턴 매칭)으로 시작합니다. 규칙 기반으로 90%를 달성하려면 케이스 예외 처리에만 시간을 다 쓰게 됩니다. 16일 안에 무리입니다. 85%를 먼저 안정적으로 달성하고 V2에서 도전하는 게 맞습니다."

**trade-off 검토**: 목표를 높게 잡으면 동기부여는 되지만, 달성하지 못했을 때 프로젝트 전체가 '실패'처럼 보이는 위험이 있습니다. 반대로 너무 낮은 목표는 성장을 설명하기 어렵습니다.

**최유나의 절충 제안**: "그럼 85%를 1차 목표로, V2(LLM 개선 단계인 DAY8 이후)에서 90% 도전을 2차 목표로 나눠 단계 목표로 설정하면 어떨까요? 그러면 V1에서 85%를 달성해도 '다음 단계 목표'가 있어서 방향이 사라지지 않습니다." → 팀 전원 동의.

**합의**: Pass Rate 85%를 1차(V1) 목표로, 90%를 2차(V2, DAY8 모델 개선 이후) 도전 목표로 이원화한다. DAY8 LLM 개선 작업과 연결된다.

**결정자**: 이도윤 (PM)

## DAY 4 · PRD 완성 + 프로토타입 검증

- **담당(lead)**: 이도윤 (PM) / 분업 시작

### 의견 조율

**주제: 선택 기능(예시 체험·Eval 표시)을 V1에 포함할 것인가**

PRD 핵심 기능 우선순위를 확정하는 과정에서 의견이 나뉘었습니다.

- **정하린 (선택 기능 포함 주장)**: "예시 문구 체험 버튼이 있으면 처음 방문한 사람도 바로 써볼 수 있어서 UX가 훨씬 풍부해집니다. Eval(Pass Rate) 표시도 서비스 신뢰도를 높여주고, 비전공 학생 발표 자료로 써도 인상적입니다. 지금 V1에 같이 넣어도 크게 어렵지 않을 것 같아요."
- **박서준 (필수 3개만 주장)**: "필수 기능 3개가 제대로 작동하는 것만으로도 16일 일정이 빡빡합니다. 선택 기능을 V1에 넣으면 필수 기능의 완성도를 올릴 시간이 줄어듭니다. Pass Rate가 85% 미달이면 서비스 자체가 의미 없어집니다. 선택 기능은 DAY 5 필수 구현이 예상보다 빨리 끝났을 때만 추가합시다."

**trade-off 검토**: 선택 기능은 UX와 신뢰도를 높이지만, 개발 시간을 가져갑니다. 필수 기능의 Pass Rate가 85% 미달인 채로 선택 기능이 들어간 서비스보다, 필수 기능이 확실히 작동하는 서비스가 더 가치 있습니다.

**합의**: 필수 3개를 먼저 구현하고, 선택 2개는 DAY 5 이후 여유가 생기면 추가한다. 선택 기능 미구현이 프로젝트 실패를 의미하지 않는다.
